## Project Overview

File Watcher is a governance enforcement subsystem for the Gov_Reg unified registry. It monitors file system events in real-time and keeps the registry synchronized with repo state. The watcher is **preventative** (real-time enforcement), not just a CI check.

## Architecture

The system follows a pipeline model: **Event Intake → Debounce/Stability → Work Queue → Governance Enforcement → Ledger Emission**.

### Core Components

- **watcher.py**: Async file watcher using `watchfiles` library with stability checking (content hash two-pass)
- **watcher_daemon.py**: Alternative daemon using `watchdog` library with `WorkQueue` and `SuppressionManager`
- **watcher_from_eafix.py**: Duplicate of watcher.py (same implementation)

Both watcher implementations use the `repo_autoops` package for models and events.

### Action Taxonomy (9 Classes)

All pipeline steps must use exactly one of these action types (defined in `ACTION_TAXONOMY_SCHEMA_V1.yaml`):
- `artifact_resolution`, `registry_query`, `template_injection`, `validation`, `computation`
- `external_execution`, `write_commit`, `state_transition`, `ledger_emission`

### Bounded Outcomes (5 States)

- `SUCCESS` (exit 0), `EXPECTED_NOOP` (exit 1), `VALIDATION_FAIL` (exit 20)
- `EXECUTION_ERROR` (exit 10), `SYSTEM_FAULT` (exit 30)

## Key Conventions

### File IDs
- Python files: `P_{20-digit}_name.py`
- Other files: `{20-digit}_name.ext`
- IDs must be exactly 20 digits starting with `01`
- Duplicate IDs are critical violations

### Registry Operations
- Single-writer semantics with file locking
- Atomic writes: temp file → validate → rename
- Backups before destructive changes at `REGISTRY\backups\`

### Stability Checking
Files must pass a content hash two-pass stability check before processing (default 2.0 second delay between hash comparisons).
