# ID Identity Contract v2.1.4

## Purpose
Define the single source of truth for all ID types, patterns, and enforcement rules in the governance system.

## ID Type Definitions

### 1. `file_id` (20-digit numeric)
- **Format**: `^\d{20}$`
- **Storage**: Always stored as 20-digit numeric string (no prefix)
- **Example**: `01999000042260125002`
- **Usage**: Primary identifier for all artifacts in the governance registry

### 2. `doc_id` (Derived, Python/Doc prefix)
- **Format**: `^P_\d{20}$`
- **Storage**: NOT stored as `file_id`; derived from `file_id` when needed
- **Derivation**: `doc_id = "P_" + file_id`
- **Example**: `P_01999000042260125002`
- **Usage**: Naming convention for Python modules and documentation artifacts

### 3. `filename_prefix` (Pattern matching)
- **Format**: `^(?:P_)?\d{20}_`
- **Purpose**: Detect ID-prefixed filenames (with or without `P_`)
- **Examples**: 
  - `01999000042260125002_canonical_id_patterns.py`
  - `P_01999000042260125002_canonical_id_patterns.py`

### 4. `enhanced_id` (22-digit extended, SEPARATE TYPE)
- **Format**: 17-digit prefix + 5-digit counter = 22 digits total
- **Structure**: `<prefix:17><counter:5>`
- **Example**: `01999000042260124000001` (17+5=22)
- **Storage**: NEVER stored as `file_id`; different identifier type
- **Usage**: Extended allocation system (if needed)
- **Important**: Must NOT be confused with `file_id`

## Canonical Sources

### Rule 1: Single Pattern Library
**ONLY** this module may define ID-matching patterns:
```
govreg_core/P_01999000042260125002_canonical_id_patterns.py
```

All other modules MUST import from this library. No local regex definitions allowed.

### Rule 2: Single Allocator Facade
**ONLY** this module may be imported for ID allocation:
```
govreg_core/P_01999000042260125006_id_allocator_facade.py
```

Backend allocator implementations are marked `INTERNAL` and blocked from direct import.

### Rule 3: Prefix-First Detection
All ID detection MUST follow this sequence:
1. Check if valid 20-digit prefix exists in filename
2. Validate pattern + uniqueness via canonical patterns module
3. Use existing prefix if valid
4. Only allocate new ID if no valid prefix found

### Rule 4: Import Restrictions (ID MODULES ONLY)
- **INTERNAL ID modules**: blocked from imports
- **DEPRECATED ID modules**: blocked from imports
- **Allocation API**: ONLY the canonical facade import surface is allowed for allocation
- **Non-ID code**: normal imports are unaffected

## Registry Storage Rules

### Registry Entry Structure
```json
{
  "file_id": "01999000042260125002",
  "relative_path": "govreg_core/P_01999000042260125002_canonical_id_patterns.py",
  "canonicality": "CANONICAL"
}
```

**Critical**: 
- `file_id` is ALWAYS 20 digits (no prefix)
- `doc_id` is NOT stored in registry; derived at read time if needed
- `canonicality` values allowed in registry: `CANONICAL`, `DEPRECATED`

```json
  "import_module": "govreg_core.P_01999000042260125002_canonical_id_patterns",
  "is_importable": true,
  "filename_prefix_kind": "DOC_PREFIXED",
  "id_generation_mode": "20_DIGIT_FILE_ID",
  "role": "PATTERN_LIBRARY",
  "canonicality": "CANONICAL",
  "superseded_by_path": null
}
```

**Canonicality Values** (inventory only):
- `CANONICAL`: The authoritative implementation
- `INTERNAL`: Backend implementation, not for direct import
- `DERIVED`: Generated/derived code
- `REFERENCE`: Kept for reference only
- `DEPRECATED`: Obsolete, superseded

## Enforcement Mechanisms

### 1. Pre-commit Gate
Blocks commits that:
- Import `DEPRECATED` or `INTERNAL` ID modules
- Define bespoke ID regex patterns outside canonical patterns module
- Define allocator-like functions without using facade
- Add new ID-prefixed scripts not in inventory

### 2. Inventory as SSOT
- All ID-related scripts tracked in `ID_SCRIPT_INVENTORY.jsonl`
- Gate reads inventory to enforce rules
- New scripts require inventory update before commit

### 3. Atomic Mutations
- Inventory updates use atomic write (temp → move)
- Evidence recorded: before/after hash, changed records, timestamp
- Idempotent operations (repeatable without side effects)

## Migration Rules

### From Legacy to Canonical
1. Identify all scripts with ID allocation/detection logic
2. Classify by role: ALLOCATOR, SCANNER, VALIDATOR, etc.
3. Mark backends as `INTERNAL`
4. Mark superseded scripts as `DEPRECATED` with `superseded_by_path`
5. Archive deprecated scripts using `git mv`
6. Update all callers to use facade

### Allowed ID Types by Context
| Context | Allowed Type | Format |
|---------|--------------|--------|
| Registry `file_id` field | `file_id` | 20 digits |
| Filename prefix | `filename_prefix` | `(?:P_)?\d{20}_` |
| Python module naming | `doc_id` | `P_\d{20}` |
| Enhanced allocation (future) | `enhanced_id` | 22 digits (17+5) |

## Success Criteria

System is drift-proof when:
1. ✅ Zero local ID regex definitions (except canonical patterns module)
2. ✅ Zero direct imports of `INTERNAL` allocators
3. ✅ Zero conflicts in `file_id` after archival
4. ✅ Gate blocks all 4 violation types (deprecated, internal, bespoke regex, new untracked)
5. ✅ Inventory freshness enforced (new ID scripts fail commit until inventoried)
