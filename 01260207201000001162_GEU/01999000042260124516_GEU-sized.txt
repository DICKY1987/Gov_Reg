Use **file_id for identity** and a **GEU registry as the SSOT for membership**, with an optional **derived column** in the file registry for convenience.

That gives you:

* stable linking (file_id doesn’t change when paths move)
* no multi-source drift (membership lives in one place)
* clean “graph” queries (GEU↔file and GEU↔GEU)

---

## 1) Canonical linking model

### A. GEU registry = SSOT (authoritative)

Each GEU entry defines a connected component:

* `geu_id` (your 16-digit GEU scheme, e.g., `99xx..............`)
* `anchor_file_id` (the file_id of the anchor artifact)
* `members[]`: list of `{file_id, role_slot, required, notes}`
* `depends_on_geu_ids[]`: inter-GEU links (optional)
* `outputs[]`: evidence/report artifacts (by file_id)
* `tests[]`: proof artifacts (by file_id)

**Key point:** membership is expressed only here (SSOT).

### B. File registry = identity inventory (authoritative for file facts)

Each file has:

* `file_id`
* `path`
* `artifact_type/role` (schema/validator/runner/etc.)
* (optional) `primary_geu_id` and/or `geu_ids[]` **but derived**, not hand-edited

---

## 2) How to link GEU files together

You have two clean options. Pick one.

### Option 1 (best): Graph edges (explicit, scalable)

Add edge records:

* `edge_type: "MEMBER_OF_GEU"`

  * `source_file_id = <file_id>`
  * `target_geu_id = <geu_id>`
  * `role_slot = "VALIDATOR" | "RUNNER" | ...`
* `edge_type: "GEU_DEPENDS_ON"`

  * `source_geu_id`
  * `target_geu_id`

This handles:

* files shared across GEUs
* cross-cutting infra
* dependency ordering between GEUs

### Option 2 (simpler): `geu_id` column on each file record

Add to each file row:

* `primary_geu_id` (single)
* `geu_ids[]` (if shared)

This is fast to query but becomes messy with shared files and is easy to let drift unless it’s generated.

**If you do Option 2, still keep the GEU registry as SSOT and generate these fields.**

---

## 3) Handling shared files (you will hit this immediately)

Some files will be used by multiple GEUs (event bus, common validators, utilities).

Do **not** force “one file → one GEU”.

Instead define ownership + consumers:

* In GEU registry:

  * one GEU lists it as `role_slot="SHARED_LIB"` (owner)
  * other GEUs reference it as `role_slot="CONSUMES_SHARED_LIB"`
* Or in edges:

  * multiple `MEMBER_OF_GEU` edges with different `role_slot`

Also add a file attribute:

* `is_shared: true`
* `owner_geu_id: <geu_id>`

---

## 4) Minimal validation rules (non-negotiable)

1. Every `geu_id` is unique.
2. Every `anchor_file_id` exists in file registry.
3. Every `members[].file_id` exists in file registry.
4. Slot matrix enforcement: GEU type requires specific roles (schema/rules/validator/runner/evidence/tests).
5. No orphan files for governance-critical roles (validators/runners/schemas must belong to ≥1 GEU or be explicitly tagged `SHARED`).

---

## 5) Merge-safety / workflow

* Store each GEU as its own file:
  `governance/geu-registry/<geu_id>.json`
* Generate:

  * `geu_index.json` (aggregated list)
  * derived `primary_geu_id/geu_ids` fields in the file registry (optional)

This avoids constant merge conflicts in one giant registry file.

---

### Bottom line

* **Link by file_id** (never by path).
* **Define GEUs in a GEU registry** (SSOT).
* **Use edges for membership + dependencies** (best long-term).
* If you want a “GEU column”, make it **generated**, not hand-edited.
