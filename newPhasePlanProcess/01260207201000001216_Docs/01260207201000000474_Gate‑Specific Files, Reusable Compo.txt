Gate‑Specific Files, Reusable Components and the New‑Gate Process
Introduction

In the new phase‑plan process the gate system provides deterministic safety checks that must be executed before advancing. A gate always consumes declared inputs, runs a prescribed set of commands, emits evidence artifacts, and produces a GateResult stating whether the gate passed or failed. Only a small adapter (the GateSpec) is different for each gate—the runner, self‑healing loop and output shapes are shared across all gates【57:0†Reusable gates and automation.txt†L1-L44】. The tables below summarise the file types involved.

1. Gate‑specific files

These are artefacts or configuration files that change per gate. Each gate has its own GateSpec file and produces its own evidence and results. The runner and schemas are not replicated; only the data changes【57:0†Reusable gates and automation.txt†L1-L44】.

File / artefact	Purpose (gate‑specific)	Notes
gates/<gate_id>.spec.json (GateSpec)	Defines the gate’s adapter. It declares the gate’s identifier, the validator to use, input paths/parameters, commands to run, the evidence contract listing required artefacts and their schema IDs, and the fingerprint fields used to compute a deterministic failure fingerprint【57:0†Reusable gates and automation.txt†L1-L44】.	This file is data‑only; it does not contain any code. Changing it customises a gate’s behaviour without modifying the runner.
Input files	Files consumed by the gate, such as a plan JSON or schema file. The GateSpec lists them in its inputs section and commands refer to these inputs【57:0†Reusable gates and automation.txt†L1-L44】.	Examples: plans/main.plan.json and its schema in the schema validation gate example【57:0†Reusable gates and automation.txt†L72-L101】.
Evidence artefacts	Files produced by the gate’s commands that form the evidence bundle. Paths and schema IDs are declared in the evidence_contract of the GateSpec【57:0†Reusable gates and automation.txt†L72-L101】. For a schema‑validation gate the evidence artefact is evidence/gates/<gate_id>/validator_report.json validated against schemas/validator_report.schema.json【57:0†Reusable gates and automation.txt†L89-L96】.	Evidence artefacts differ by gate because each validator produces different outputs; they are validated against schemas to guarantee shape and allow deterministic replay.
results/<gate_id>/GateResult.json	Output generated by the shared runner. Contains the gate’s identifier, validator, boolean passed flag, exit code, normalized error codes and a failure fingerprint【57:0†Reusable gates and automation.txt†L20-L27】.	Each gate run produces its own GateResult.json; however the schema is shared and stable across all gates【57:0†Reusable gates and automation.txt†L37-L44】.
results/<gate_id>/HealSequenceReport.json (optional)	Produced only when remediation is invoked. Records the ordered heal templates executed, whether healing succeeded, and the final verdict【57:0†Reusable gates and automation.txt†L28-L35】.	Like the GateResult, the report shape is fixed; only values vary by gate.
Explanation

GateSpec is the only gate‑specific configuration. The process restricts customisation to five fields: input shape, command recipe, evidence extractor, evidence schemas and binary pass criteria【69:0†Reusable gates and automation.txt†L50-L58】. Everything else—including how the gate is executed, evidence is validated, results are emitted and healing is orchestrated—is handled by shared components【57:0†Reusable gates and automation.txt†L1-L44】.

Evidence is produced according to the evidence_contract and must validate against its schema. This ensures that downstream consumers can rely on consistent shapes even though the contents differ per gate【57:0†Reusable gates and automation.txt†L89-L96】. Evidence bundles are referenced in the GateResult via evidence_paths【57:0†Reusable gates and automation.txt†L20-L27】.

GateResult and HealSequenceReport have the same structure for every gate, allowing them to be consumed by generic tooling (metrics, audit, gating orchestrator)【57:0†Reusable gates and automation.txt†L37-L44】.

2. Reusable files and shared components

The following files or modules are shared across all gates. They define schemas, orchestrators and helper functions that remain unchanged when gates are added or modified.

File / module	Purpose (shared)	Notes
schemas/gate_spec.schema.json	JSON Schema defining the structure of a GateSpec. Ensures that every GateSpec contains the allowed fields—gate_id, validator_id, inputs, commands, evidence_contract, pass_criteria and fingerprint_contract—and disallows undeclared fields【57:0†Reusable gates and automation.txt†L1-L44】.	This schema enforces the “data‑only” principle and prevents accidental code or unreviewed parameters from creeping into gate configurations.
schemas/gate_result.schema.json	Schema describing the canonical GateResult. It validates that every result has the standard fields (gate identifier, validator id, boolean passed, exit code, normalized error codes, failure fingerprint and evidence paths)【57:0†Reusable gates and automation.txt†L20-L27】.	Because the runner emits identical shapes for every gate, analysis tools can process GateResult files without knowing which gate produced them【57:0†Reusable gates and automation.txt†L37-L44】.
schemas/heal_sequence_report.schema.json	Schema for the optional HealSequenceReport. It validates the presence of fields such as healed, ordered heal_template_ids and the final verdict【57:0†Reusable gates and automation.txt†L28-L35】.	Standardizing this output allows automated systems to interpret remediation outcomes uniformly.
schemas/heal_template_result.schema.json	Schema used by heal templates to record the result of each remediation attempt.	Shared across gates; not present in the GateSpec but used by the self‑heal orchestrator【69:0†Reusable gates and automation.txt†L82-L90】.
schemas/failure_mode.schema.json	Schema defining a FailureMode object. Failure modes classify failure fingerprints and associate them with ordered heal sequences.	New failure modes can be added without changing code【69:0†Reusable gates and automation.txt†L61-L69】.
schemas/validator_report.schema.json	Evidence schema used by many gates (e.g., schema‑validation). Validates the structure of validator reports so that GateRunner can confirm that evidence matches expectations【57:0†Reusable gates and automation.txt†L89-L96】.	Additional evidence types require their own schemas but they are reusable across gates.
scripts/gate_runner.py	Implementation of the shared GateRunner. Performs the invariant pipeline for any gate: acquires locks, runs the commands, normalizes raw outputs to exit codes and normalized error codes, validates evidence against schemas, emits GateResult.json and computes the failure fingerprint【57:0†Reusable gates and automation.txt†L1-L44】.	No gate contains its own runner; all gates use this shared script to enforce determinism and to avoid bespoke code.
scripts/self_heal_orchestrator.py	Implementation of the shared remediation loop. When a gate fails, it classifies the failure using the configured failure modes, executes heal templates in order, records a HealSequenceReport and re‑runs the gate if healing succeeds【57:0†Reusable gates and automation.txt†L28-L35】.	Because healing is centralized, gates cannot implement their own retry logic, ensuring that remediation remains deterministic【69:0†Reusable gates and automation.txt†L61-L69】.
scripts/error_normalizer.py	Normalizes raw tool outputs into canonical error codes according to the mapping provided in each GateSpec (via error_code_normalization). This supports the fingerprint contract and ensures consistent error taxonomies【69:0†Reusable gates and automation.txt†L50-L58】.	Shared across gates; only the mapping values change per gate.
configs/failure_modes.json	Collection of FailureMode objects. Each failure mode contains a matcher to recognise a failure fingerprint and an ordered list of heal templates to run【69:0†Reusable gates and automation.txt†L61-L69】.	When adding gates you typically extend this file to teach the self‑heal system how to react to new failure fingerprints.
heal_templates/<template_id>.yaml	Declarative heal templates referenced by failure modes. They describe how to remediate specific failure types.	Because templates are shared and referenced by ID, multiple gates can reuse the same heal logic【69:0†Reusable gates and automation.txt†L61-L69】.
gate_catalog.json	List of all gates in the system and their metadata (e.g., descriptions, category, phase). Used by orchestrators to discover gates.	Updating this file registers a new gate so it can be invoked by the runner.
scripts/gate_dependencies.json	Defines the dependency graph or execution order of gates. The gate execution engine loads this file to compute a topological order【57:0†Reusable gates and automation.txt†L1-L44】.	When a new gate is added you must add its dependencies to this file.
schemas/gate_catalog.schema.json	Schema that validates the structure of the gate catalogue.	Part of the phase contract example; ensures the catalogue is well‑formed【57:0†Reusable gates and automation.txt†L1-L44】.
metrics/metrics_report.py	Script that reads GateResults and evidence bundles to produce aggregated metrics about gate performance.	Because GateResult structures are consistent, this script works across all gates.
Why these files are reusable

The reusable components encapsulate all logic that should be identical across gates. GateRunner and SelfHealOrchestrator implement the deterministic pipeline—run, validate, optionally heal—that every gate must follow【57:0†Reusable gates and automation.txt†L1-L44】. Schemas such as gate_spec.schema.json, gate_result.schema.json and heal_sequence_report.schema.json codify the shapes of inputs and outputs to prevent drift. Shared configuration files (failure_modes.json, gate_dependencies.json) centralise data that would otherwise lead to scattered logic. As a result, when you add or modify a gate you change only data, not code【69:0†Reusable gates and automation.txt†L50-L58】.

3. Files involved in adding a new gate (end‑to‑end process)

When creating a fully functioning reusable gate, the process focuses on producing or modifying data files so that the shared runner and self‑heal logic can execute the gate without bespoke scripts. The key steps and the files they touch are listed below.

Define the gate’s purpose and confirm it is a binary check. Gates must have a pass/fail outcome and must be able to produce deterministic evidence【57:0†Reusable gates and automation.txt†L1-L44】.

Create a new GateSpec file. Add gates/<new_gate_id>.spec.json containing:

gate_id and validator_id to uniquely identify the gate【57:0†Reusable gates and automation.txt†L1-L44】.

inputs listing the input files or identifiers the gate will consume【57:0†Reusable gates and automation.txt†L1-L44】.

commands describing how to run the validator tool【57:0†Reusable gates and automation.txt†L1-L44】.

evidence_contract enumerating required evidence artefacts, their paths and associated schema IDs【57:0†Reusable gates and automation.txt†L72-L101】.

fingerprint_contract listing the fields used to compute the failure fingerprint【57:0†Reusable gates and automation.txt†L72-L101】.

Optionally error_code_normalization to map raw tool outputs to canonical error codes【69:0†Reusable gates and automation.txt†L50-L58】.

Define or extend evidence schemas. For every artefact in evidence_contract that is not already covered, create a new JSON Schema under schemas/ (e.g., schemas/new_validator_report.schema.json). These schemas allow the runner to validate evidence and to fail closed when evidence does not match expectations【57:0†Reusable gates and automation.txt†L89-L96】.

Extend failure modes and heal templates if required. If the new gate can fail in ways not captured by existing failure modes, append new objects to configs/failure_modes.json. Each new FailureMode object includes a matcher for the failure fingerprint and an ordered list of heal template IDs【69:0†Reusable gates and automation.txt†L61-L69】. If you need new remediation logic, add heal templates in heal_templates/ and reference them from the failure mode.

Register the gate. Update gate_catalog.json to list the new gate and its metadata (e.g., description, phase). Update scripts/gate_dependencies.json to include the new gate in the dependency graph. This ensures the orchestrator knows when to execute the gate and in what order.

Run the gate with the shared runner. No additional scripts are required. The existing gate_runner.py will read the new GateSpec, run the commands, produce evidence, validate schemas, emit GateResult.json and, if necessary, invoke the self‑heal orchestrator【57:0†Reusable gates and automation.txt†L1-L44】.

Verify outputs. Confirm that the new gate produces a valid GateResult and (when invoked) a HealSequenceReport. These files live under results/<new_gate_id>/ and conform to their respective schemas【57:0†Reusable gates and automation.txt†L37-L44】.

Integrate into higher‑level plans. If the new gate is part of a phase, ensure that the phase contract lists the new gate_results_bundle among its required or produced artefacts so that downstream steps expect it【57:0†Reusable gates and automation.txt†L1-L44】.

Summary

The gate framework deliberately separates data from execution logic. Gate‑specific files (GateSpec, evidence artefacts and results) allow each gate to implement a unique check without introducing custom code. Reusable files (schemas, runners, orchestrators and configuration files) encapsulate all common behaviour, enforcing determinism and auditability. When adding a new gate, engineers create or modify only data files—primarily the GateSpec, evidence schemas and failure‑mode definitions—while relying on existing scripts and schemas for execution【69:0†Reusable gates and automation.txt†L50-L58】. This approach reduces duplication, makes gate behaviour predictable and simplifies integration into larger phase‑plan processes.
