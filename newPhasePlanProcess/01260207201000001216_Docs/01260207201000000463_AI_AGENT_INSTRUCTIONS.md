## CRITICAL RULES - MUST FOLLOW

### ❌ GATE-000 WILL FAIL IF YOU:
- Use TODO, TBD, FIXME, or any placeholders
- Use "..." or "etc." or vague descriptions
- Include comments like "fill this in later"
- Use variables like {PLACEHOLDER} or <FILL_IN>

**Every field MUST have real, specific content.**

### ❌ GATE-003 WILL FAIL IF YOU:
- Any step is missing: inputs, outputs, preconditions, postconditions, commands, rollback, evidence, idempotency
- Step IDs are not unique
- Commands are descriptions instead of executable shell commands

**EVERY step needs ALL 8 contract fields.**

### ❌ GATE-014 WILL FAIL IF YOU:
- An artifact is produced but never consumed (orphan)
- An artifact is consumed but never produced (dead)
- Artifact IDs don't match between steps and artifacts section

**All artifact flows must CLOSE - every input needs a producer, every output needs a consumer.**

### ❌ GATE-015 WILL FAIL IF YOU:
- Any step is missing rollback procedures
- Rollback strategy is not one of: delete, restore, revert, manual

**EVERY step must be reversible.**

### ❌ GATE-016 WILL FAIL IF YOU:
- Any step is missing postconditions
- Postconditions don't have evidence_path
- Evidence paths don't follow pattern: .state/evidence/{phase_id}/{step_id}/

**EVERY step must have verification.**

### ❌ GATE-017 WILL FAIL IF YOU:
- Same step_id appears twice
- Same artifact_id appears twice
- Same assumption_id appears twice

**All IDs must be unique.**

### ❌ FM-06 WILL FAIL IF YOU:
- Phase dependencies form a cycle (PH-001 → PH-002 → PH-001)

**Dependencies must form a DAG (Directed Acyclic Graph).**

## STEP-BY-STEP GENERATION PROCESS

### Step 1: Analyze the Project Request
- Identify the project goal
- Determine complexity: simple (1-2 phases), moderate (3-5 phases), complex (6+ phases)
- List all inputs and outputs
- Identify assumptions and unknowns

### Step 2: Define Phases
- Break work into logical phases (Setup, Build, Test, Deploy, etc.)
- Each phase should have 1-10 steps
- Phases must be sequential or explicitly parallel

### Step 3: Define Steps (FOR EACH PHASE)
- Give each step a unique ID: ST-001, ST-002, etc.
- Define what files/data this step needs (inputs)
- Define what files/data this step creates (outputs)
- Write preconditions (what must be true before)
- Write exact shell commands (not descriptions)
- Write postconditions (what must be true after)
- Write rollback commands (how to undo)
- Specify evidence paths

### Step 4: Map Artifacts
- Every input → must have a producer (another step's output or external)
- Every output → must have a consumer (another step's input or final deliverable)
- Create artifacts section with producer_step and consumer_steps

### Step 5: Document Assumptions
- List what you're assuming (e.g., "Python 3.11 is installed")
- Describe risk if false
- Specify how to validate

### Step 6: Build Navigation DAG
- List all phase_ids in nodes array
- Create edges showing dependencies [[PH-001, PH-002] means PH-002 depends on PH-001]
- Create dependencies object showing what each phase depends on

### Step 7: Validate Mentally
- Check: No TODO/TBD/FIXME?
- Check: Every step has all 8 contract fields?
- Check: All artifacts close (inputs have producers, outputs have consumers)?
- Check: No circular dependencies?
- Check: Commands are executable (not descriptions)?

```json
  }
```

### ✅ GOOD - All Artifacts Close
```json
"steps": [
  {
    "step_id": "ST-001",
    "inputs": [],
    "outputs": [{"artifact_ref": "ART-001", "path": "data.json"}]
  },
  {
    "step_id": "ST-002",
    "inputs": [{"artifact_ref": "ART-001", "path": "data.json"}],  // Consumes ART-001
    "outputs": [{"artifact_ref": "ART-002", "path": "report.pdf"}]
  }

"artifacts": {
  "ART-001": {
    "producer_step": "ST-001",
    "consumer_steps": ["ST-002"]  // Consumed
  },
  "ART-002": {
    "producer_step": "ST-002",
    "consumer_steps": []  // Final deliverable - OK to have no consumers
  }
}
```
