## What This Is

newPhasePlanProcess is a deterministic plan validation and execution system. Plans are JSON documents conforming to a v2.4.0 schema (JSON Schema Draft 2020-12) with three required top-level sections: `meta`, `plan`, and `navigation`. The system validates plans through a dependency-ordered pipeline of validation gates and failure mode checks, producing timestamped JSON evidence artifacts.

## Commands

### Plan CLI (`scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py`)

```bash
# Validate plan against schema
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py validate <plan.json>

# Normalize structure (deterministic ordering)
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py normalize <plan.json> [output.json]

# Compile (normalize + regenerate navigation DAG + validate formulas)
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py compile <plan.json> <output.json>

# Execute plan
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py execute <plan.json>

# Diff two plans (ignoring timestamps)
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py diff <plan_a.json> <plan_b.json>

# Run all validation gates in dependency order
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py run-gates <plan.json> --evidence-dir .state/evidence
python scripts/P_01260207233100000242_NEWPHASEPLANPROCESS_plan_cli.py run-gates --plan-file <plan.json> --fail-fast --only-required
```

## Architecture

### Gate Validation Pipeline

Gates execute in dependency order defined in `01260207233100000678_gate_dependencies.json`. The critical path is:

```
GATE-000 → GATE-001 → GATE-002 → GATE-010 → GATE-011 → GATE-012 → GATE-013
```

Gates are grouped into execution phases:
- **pre-validation**: GATE-000 (structure check, obfuscation detection)
- **core-validation**: GATE-001 (schema), GATE-002 (gate definitions), GATE-005 (planning artifacts)
- **automation**: GATE-006 → GATE-007 → GATE-008 (spec → index → diagrams)
- **wiring**: GATE-010 → GATE-011 → GATE-012 → GATE-013 (proof → registry → FM audit → E2E)
- **meta-validation**: GATE-998 (automation audit), GATE-999 (goal reconciliation)
- **optional**: VAL-FILE-MANIFEST, VAL-FILE-SCOPE, VAL-CI-COMPAT, VAL-GROUND-TRUTH, VAL-METRICS, etc.

Gates with the same `depends_on` can run in parallel. If a required gate fails, its dependents should not run.

### Failure Modes (FM-01 through FM-12)

Twelve wiring integrity checks in `scripts/wiring/`, each detecting a specific integration failure:

| FM | Check | Script |
|----|-------|--------|
| FM-01 | Orphaned artifacts (produced, never consumed) | `P_01260207233100000277_check_orphans.py` |
| FM-02 | Write conflicts (multiple writers to same artifact) | `P_01260207233100000279_detect_write_conflicts.py` |
| FM-03 | Incomplete handoffs | `P_01260207233100000285_validate_handoffs.py` |
| FM-04 | Dead artifacts (consumed, never produced) | `P_01260207233100000274_check_dead_artifacts.py` |
| FM-05 | Missing producers | `P_01260207233100000276_check_missing_producers.py` |
| FM-06 | Circular dependencies | `P_01260207233100000278_detect_cycles.py` |
| FM-07 | Dormant flows (defined but never triggered) | `P_01260207233100000275_check_dormant_flows.py` |
| FM-08 | Evidence bundle gaps | `P_01260207233100000284_validate_evidence_bundles.py` |
| FM-09 | Recovery policy gaps | `P_01260207233100000286_validate_recovery_policies.py` |
| FM-10 | Idempotency violations | `P_01260207233100000281_test_idempotency_all.py` |
| FM-11 | Detection gate gaps | `P_01260207233100000282_validate_detection_gates.py` |
| FM-12 | E2E proof linkage gaps | `P_01260207233100000283_validate_e2e_proof_linkage.py` |

All FM checks run collectively via GATE-012 (`scripts/wiring/P_01260207233100000273_audit_failure_modes.py`).

### Evidence System

Every gate produces JSON evidence artifacts under `.state/evidence/GATE-###/`. Evidence includes timestamps, pass/fail status, and structured error/warning objects. Exit code 0 = pass, 1 = fail.

### Key Classes in the CLI

- **SafeFormulaEvaluator** - Arithmetic expression evaluator (no `eval()`); supports `+`, `-`, `*`, `/`, `MIN`, `MAX`, `CEIL`, `FLOOR`, and variables
- **PlanValidator** - Validates constraints beyond JSON Schema: ID uniqueness, referential integrity, dependency acyclicity, command references, scheduling validity
- **PlanExecutor** - Reads only `/plan` and `/meta` sections for execution

### Converter Scripts

- `P_01260207233100000233_DOC-CORE-PHASE-PLAN-TO-WORKSTREAM-CONVERTER-PY-001__phase_plan_to_workstream_converter.py` - Phase ↔ workstream bidirectional

## Constraints

- **No eval()**: The CLI uses SafeFormulaEvaluator. GATE-000 blocks `chr()`, `eval()`, `exec()`, `base64` obfuscation patterns.
- **Determinism**: Same input must produce byte-identical output. Arrays are sorted, ID-keyed maps canonical, hashes stable.
- **Single-writer rule**: Each artifact must have exactly one producer (FM-02, FM-05 enforce this).
- **Schema location**: `schemas/01260207233100000674_NEWPHASEPLANPROCESS_plan.schema.v2.4.0.json` - the CLI resolves it relative to its own `__file__` path.
- **Evidence immutability**: Evidence directories should be created fresh for each run (`--evidence-dir` controls output location).
