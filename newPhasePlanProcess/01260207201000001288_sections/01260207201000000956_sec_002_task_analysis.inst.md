# SEC-002: Task Analysis + Classification Instructions

**Schema File:** `sections/sec_002_task_analysis.json`  
**Section ID:** `sec_002_task_analysis`  
**Required:** Yes  
**Updated:** 2026-01-28

---

## What To Do

1. Capture the original user request verbatim
2. Transform vague requirements into **mechanically verifiable** targets
3. Classify the project complexity using the decision tree
4. Log the classification decision with rationale

---

## How To Do It

### Step 1: Capture Original Request

Copy the user's request **exactly as given**:
```json
{
  "original_request": {
    "verbatim": "Build a CLI tool to validate JSON schemas",
    "timestamp": "2026-01-28T06:09:17.948Z",
    "source": "user_input"
  }
}
```

**Source options:** `user_input`, `github_issue`, `slack_thread`, `other`

### Step 2: Transform Vague → Quantified

For EVERY vague requirement, create a quantified target with validation command:

**Template:**
```json
{
  "vague": "{{ORIGINAL_VAGUE_STATEMENT}}",
  "quantified": "{{MEASURABLE_TARGET}}",
  "validation": "{{EXECUTABLE_COMMAND}}",
  "expect_regex": "/{{PATTERN}}/"
}
```

**Common transformations:**

| Vague | Quantified | Validation Command |
|-------|------------|-------------------|
| "Fast" | "<100ms p95 latency" | `python benchmark.py \| grep p95` |
| "Reliable" | "100% tests pass" | `pytest --tb=short -q` |
| "Well-tested" | ">95% coverage" | `pytest --cov=src \| grep TOTAL` |
| "Secure" | "0 high/critical vulns" | `bandit -r src/ -ll` |
| "Clean code" | "0 lint errors" | `ruff check src/` |
| "Type-safe" | ">90% type coverage" | `mypy src/ --strict` |

**Create 5-10 quantified targets** covering:
- Performance (latency, throughput)
- Reliability (test pass rate, uptime)
- Quality (coverage, lint score, type coverage)
- Security (vulnerability scan)
- Maintainability (doc coverage, complexity metrics)

### Step 3: Run Classification Decision Tree

Analyze the request to count:
- `file_count`: Estimated number of files to create/modify
- `loc`: Estimated lines of code
- `requires_integration_tests`: true/false
- `requires_contracts`: true/false (multiple services/APIs)

**Apply decision tree rules in order:**

```
IF file_count <= 3 AND loc < 200
  → CLASSIFY-SIMPLE
  → verification_level: 1, test_coverage_min: 70%
  → phases: [intake, implement, verify]

ELSE IF file_count <= 10 AND loc < 1000
  → CLASSIFY-MODERATE
  → verification_level: 2, test_coverage_min: 80%
  → phases: [intake, plan, implement, verify]

ELSE IF file_count <= 30 OR requires_integration_tests
  → CLASSIFY-COMPLEX
  → verification_level: 2, test_coverage_min: 90%
  → phases: [intake, architecture, implement, integration, verify]

ELSE (file_count > 30 OR requires_contracts)
  → CLASSIFY-ENTERPRISE
  → verification_level: 3, test_coverage_min: 95%
  → phases: [bootstrap, architecture, contracts, implement_parallel, integration, hardening, package]
```

### Step 4: Set Classification Fields

Based on decision tree result:
```json
{
  "classification": {
    "complexity": "moderate",
    "domain": "technical",
    "quality": "standard",
    "inferred_behavior": {
      "verification_level": 2,
      "test_coverage_min": 80,
      "phases": ["intake", "plan", "implement", "verify"],
      "parallel_groups": []
    }
  }
}
```

**Domain options:**
- `technical` - Code/engineering work
- `analytical` - Data analysis/research
- `creative` - Design/content creation
- `general` - Mixed or unclear

**Quality options:**
- `standard` - Normal quality bar
- `high` - Elevated quality requirements
- `critical` - Safety/security critical
- `production` - Production-grade system

### Step 5: Log Classification Decision

Record the decision in SEC-014 (Decision Ledger):
```json
{
  "decision_id": "DEC-CLASSIFY-001",
  "category": "classification",
  "context": {
    "file_count": 8,
    "loc": 500,
    "requires_contracts": false,
    "requires_integration_tests": false
  },
  "options": ["simple", "moderate", "complex", "enterprise"],
  "selected_option": "moderate",
  "rationale": "File count (8) and LOC (500) fall within moderate range per decision tree rule CLASSIFY-MODERATE",
  "timestamp": "2026-01-28T06:09:17.948Z",
  "owner": "AI_AGENT"
}
```

---

## Validation Rules

1. `original_request.verbatim` MUST be non-empty
2. `timestamp` MUST be ISO 8601 format
3. MUST have 5-10 specificity transformations
4. Each transformation MUST have: vague, quantified, validation, expect_regex
5. `complexity` MUST be one of: simple, moderate, complex, enterprise
6. `verification_level` MUST match complexity (simple=1, moderate/complex=2, enterprise=3)
7. Decision ledger entry MUST be created

## Examples

### Example 1: CLI Tool (Moderate)
**Input:** "Build a CLI tool to validate JSON schemas"

**Analysis:**
- Estimated files: 6-8 (cli.py, validator.py, schema_loader.py, tests/)
- Estimated LOC: 400-600
- No integration tests needed
- No contracts needed

**Classification:** MODERATE
```json
{
  "classification": {
    "complexity": "moderate",
    "domain": "technical",
    "quality": "standard",
    "inferred_behavior": {
      "verification_level": 2,
      "test_coverage_min": 80,
      "phases": ["intake", "plan", "implement", "verify"],
      "parallel_groups": []
    }
  }
}
```

**Quantified targets:**
1. "Works correctly" → "100% of tests pass" | `pytest --tb=short -q` | exit_code=0
2. "Fast" → "<10ms validation time" | `time python validate.py sample.json` | /real.*0\.00[0-9]s/
3. "Well-tested" → ">80% coverage" | `pytest --cov=src` | /TOTAL.*8[0-9]|9[0-9]|100%/
4. "Clean code" → "0 lint errors" | `ruff check src/` | exit_code=0
5. "User-friendly" → "Help text exists" | `python cli.py --help` | /Usage:/

### Example 2: Microservice Architecture (Enterprise)
**Input:** "Build a microservices platform with 5 services, API gateway, and message queue"

**Analysis:**
- Estimated files: 50+ (multiple services)
- Estimated LOC: 5000+
- Requires integration tests (service interactions)
- Requires contracts (API contracts between services)

**Classification:** ENTERPRISE

---

## Common Mistakes

1. **Forgetting to quantify** - Every target must be measurable
2. **Missing validation commands** - Every target needs a command
3. **Wrong complexity level** - Follow decision tree strictly
4. **Skipping decision ledger** - Always log the classification decision
5. **Vague quantifications** - "Fast" is not quantified; "<100ms" is

---

## AI Agent Notes

- Classification drives the entire plan structure
- Higher complexity → more phases, gates, evidence
- Don't overthink file counts - estimate conservatively
- When in doubt between two levels, choose higher complexity
