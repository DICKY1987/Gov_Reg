# SEC-000: Critical Constraints Instructions

**Schema File:** `sections/sec_000_critical_constraints.json`  
**Section ID:** `sec_000_critical_constraints`  
**Required:** Yes  
**Updated:** 2026-01-28

---

## What To Do

Enforce the **NO IMPLIED BEHAVIOR** constraint and **DETERMINISM** constraint across the entire plan. This section defines non-negotiable rules that must be validated before plan execution.

---

## How To Do It

### Step 1: Understand the Constraints

**NO IMPLIED BEHAVIOR:**
- Every validation gate, rule, and contract MUST be expressed as:
  1. A config entry
  2. An executable command
  3. An evidence artifact path
  4. A pass/fail condition (regex or exit code)

**DETERMINISM:**
- Plans must achieve 95%+ reproducibility
- Same inputs → Same outputs across multiple runs

### Step 2: Mark Non-Automatable Items

If you encounter ANY item that cannot be expressed as config+command+evidence:
- Mark it with `[NOT_AUTOMATABLE]`
- Escalate with explicit human approval gate
- Document WHY it cannot be automated

### Step 3: Validate Against Forbidden Patterns

Scan the entire plan and flag these patterns:
- ❌ "works well", "good enough" (vague success criteria)
- ❌ "looks correct", "seems fine" (subjective pass/fail)
- ❌ "fix it" without specific action (open-ended)
- ❌ TODO, TBD, placeholder code in deliverables
- ❌ Skeleton/stub implementations in final output

### Step 4: Ensure All Gates Are Mechanical

For EVERY validation gate in the plan:
```
✓ Has executable command
✓ Has expected exit code
✓ Has regex pattern for stdout/stderr
✓ Has evidence output path
✓ Has timeout value
```

### Step 5: Validate Determinism Requirements

Ensure these exist elsewhere in the plan:
- Decision ledger with ALL decisions logged
- Assumption ledger with validation methods
- File scope enforcement (hard boundaries)
- Ground truth levels (L0-L5) with mechanical pass/fail
- DAG-scheduled parallel execution
- Bounded retry logic with convergence detection

---

## Validation Rules

1. **validation_command** must exist and be executable
2. **expect_exit_code** must be 0 for success validation
3. **expect_stdout_regex** must contain at least one pattern
4. **target_reproducibility** must be >= 0.95
5. All six determinism requirements must be satisfied by other sections

## Examples

### Example 1: Valid Mechanical Gate
```json
{
  "gate_id": "GATE-001",
  "command": "pytest tests/ -v --tb=short",
  "expect_exit_code": 0,
  "expect_stdout_regex": ["/PASSED/", "/100% coverage/"],
  "evidence_path": ".runs/test_evidence.log",
  "timeout_seconds": 300
}
```

### Example 2: Invalid (Vague) Gate
```json
{
  "gate_id": "GATE-BAD",
  "command": "manually review code quality",
  "expect": "looks good"
}
```
❌ This violates NO IMPLIED BEHAVIOR - must be marked [NOT_AUTOMATABLE] with justification.

---

## Common Mistakes

1. **Forgetting to mark manual steps** - Always add `[NOT_AUTOMATABLE]` tag
2. **Subjective success criteria** - "Good enough" is not mechanical
3. **Missing evidence paths** - Every command must write evidence
4. **No timeout** - All commands must have bounded execution time
5. **Implied behavior** - "Run tests" without specifying which tests/command
