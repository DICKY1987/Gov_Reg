# SEC-004: Assumptions & Scope Contract Instructions

**Schema File:** `sections/sec_004_assumptions_scope.json`  
**Section ID:** `sec_004_assumptions_scope`  
**Required:** Yes  
**Updated:** 2026-01-28

---

## What To Do

1. Document ALL assumptions with validation methods
2. Define clear scope boundaries (in/out/non-goals)
3. State constraints (time, security, performance, regulatory)
4. Define global "definition of done" criteria

---

## How To Do It

### Step 1: Build Assumptions Ledger

For EVERY assumption in the plan, create an entry:

**Template:**
```json
{
  "assumption_id": "ASM-001",
  "statement": "Python 3.11+ is available on target systems",
  "impact_score": 8,
  "confidence": "HIGH",
  "validation_method": "Check Python version in CI",
  "validation_command": "python --version | grep 'Python 3.1[1-9]'",
  "expect_exit_code": 0,
  "expect_stdout_regex": ["/Python 3\\.1[1-9]/"],
  "not_automatable": false
}
```

**Impact score (1-10):**
- 9-10: Critical (plan fails if false)
- 7-8: High (major rework needed if false)
- 4-6: Medium (some rework needed if false)
- 1-3: Low (minor adjustments if false)

**Confidence:**
- HIGH: 90%+ certain assumption is true
- MEDIUM: 50-90% certain
- LOW: <50% certain (flag for validation)

**Common assumption categories:**
- Environment: OS, Python version, available tools
- Dependencies: Libraries, services, APIs available
- Data: File formats, schemas, data availability
- Access: Permissions, credentials, network access
- User behavior: Expected inputs, usage patterns

**Validation command MUST:**
- Be executable in CI/local environment
- Return 0 on success, non-zero on failure
- Write evidence to `.state/assumptions.jsonl`

**If truly not automatable:**
```json
{
  "assumption_id": "ASM-MANUAL-001",
  "statement": "Stakeholder approves UI design",
  "not_automatable": true,
  "validation_method": "[NOT_AUTOMATABLE] Requires human design review meeting"
}
```

Aim for **10-20 assumptions** - be thorough.

### Step 2: Define Scope Contract

#### In Scope (5-10 items)
Be **specific and concrete**:
- ✅ "Validate JSON files against JSON Schema draft-07"
- ✅ "Support schemas up to 10MB in size"
- ✅ "CLI interface with --file and --schema arguments"
- ❌ "Good validation" (vague)

#### Out of Scope (5-10 items)
Explicitly call out what's **excluded**:
- "No GUI/web interface"
- "No support for YAML schemas"
- "No schema generation from examples"
- "No cloud deployment"

#### Non-Goals (3-5 items)
Higher-level things we're **intentionally not solving**:
- "Not a general-purpose JSON parser"
- "Not a schema editor or IDE"
- "Not a schema registry service"

### Step 3: Define Constraints

Only fill if applicable:

```json
{
  "constraints": {
    "time_budget": "OPTIONAL - only if hard deadline exists",
    "security_compliance": "Must pass OWASP Top 10 checks",
    "performance_targets": "<100ms p95 latency",
    "regulatory": "GDPR compliant (no PII storage)"
  }
}
```

Leave as `{{OPTIONAL}}` if not applicable.

### Step 4: Define Global Definition of Done

Write **6-8 criteria** that apply to the ENTIRE plan:

**Standard criteria (always include):**
1. "Outputs are schema-valid (where schemas exist)"
2. "Deterministic outputs stable across 3 runs"
3. "Unit + integration tests pass (100%)"
4. "All 'must_pass' validations pass in every phase contract"
5. "Audit/evidence logs exist at declared locations"

**Add 1-3 project-specific criteria:**
- "CLI tool installed and runnable"
- "Documentation covers 80%+ of public APIs"
- "Security scan shows 0 high/critical issues"

---

## Validation Rules

1. MUST have ≥10 assumptions
2. Each assumption MUST have: statement, impact_score (1-10), validation_method
3. Automatable assumptions MUST have: validation_command, expect_exit_code, expect_stdout_regex
4. Not-automatable assumptions MUST have: not_automatable=true
5. In-scope MUST have 5-10 items
6. Out-of-scope MUST have 5-10 items
7. Global DoD MUST have 6-8 items

## Examples

### Example 1: High-Impact Assumption
```json
{
  "assumption_id": "ASM-001",
  "statement": "Target systems have Python 3.11 or higher",
  "impact_score": 9,
  "confidence": "HIGH",
  "validation_method": "Check Python version on CI and local",
  "validation_command": "python --version",
  "expect_exit_code": 0,
  "expect_stdout_regex": ["/Python 3\\.1[1-9]/"],
  "not_automatable": false
}
```

### Example 2: Not Automatable
```json
{
  "assumption_id": "ASM-005",
  "statement": "User has basic understanding of JSON Schema spec",
  "impact_score": 4,
  "confidence": "MEDIUM",
  "validation_method": "[NOT_AUTOMATABLE] Requires user documentation and examples",
  "not_automatable": true
}
```

### Example 3: Complete Scope Contract
```json
{
  "scope_contract": {
    "in_scope": [
      "Validate JSON files against JSON Schema draft-07 and draft-2020-12",
      "CLI interface with --file, --schema, --strict flags",
      "Support schemas up to 10MB in size",
      "Output validation errors in JSON format",
      "Exit code 0 for valid, 1 for invalid, 2 for errors"
    ],
    "out_of_scope": [
      "No GUI or web interface",
      "No YAML schema support",
      "No schema generation from JSON examples",
      "No schema editing capabilities",
      "No cloud/SaaS deployment"
    ],
    "non_goals": [
      "Not a general-purpose JSON parser library",
      "Not a schema registry or management system",
      "Not a schema documentation generator"
    ]
  }
}
```

## Common Mistakes

1. **Too few assumptions** - Capture EVERYTHING you're assuming
2. **Missing validation commands** - Every assumption needs verification
3. **Vague scope** - "Handle JSON" is not specific enough
4. **Forgetting "out of scope"** - Explicit exclusions prevent scope creep
5. **Weak DoD criteria** - "Good quality" is not measurable

## AI Agent Notes

- Assumptions are often **implicit** - make them explicit
- Higher impact scores → validate earlier in plan
- Out-of-scope is as important as in-scope
- This section prevents future "I thought you meant..." issues
