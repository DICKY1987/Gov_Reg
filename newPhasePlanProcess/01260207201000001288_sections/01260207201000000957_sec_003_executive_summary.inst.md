# SEC-003: Executive Summary Instructions

**Schema File:** `sections/sec_003_executive_summary.json`  
**Section ID:** `sec_003_executive_summary`  
**Required:** Yes  
**Updated:** 2026-01-28

---

## What To Do

Create a decision-ready executive summary with:
1. One-sentence outcome statement
2. What changes vs. what doesn't change
3. Primary decisions needed (blocking questions)
4. Top 3 risks with mitigation
5. High-level "done" criteria

---

## How To Do It

### Step 1: Write One-Sentence Outcome

Synthesize the entire plan into ONE sentence:

**Template:** "Deliver {WHAT} that {KEY_CAPABILITY} with {MEASURABLE_OUTCOME}"

**Examples:**
- "Deliver CLI tool that validates JSON schemas with 100% test coverage and <10ms latency"
- "Deploy FastAPI service to AWS that handles 1000 req/sec with 99.9% uptime"
- "Refactor authentication module to support OAuth2 with zero downtime migration"

### Step 2: List What Changes

Be **specific and concrete**:
- ✅ "Create src/validator.py with schema validation logic"
- ✅ "Modify tests/test_api.py to add 15 new integration tests"
- ✅ "Update config/settings.yaml with new OAuth2 parameters"
- ❌ "Improve code quality" (too vague)
- ❌ "Refactor some files" (not specific)

Aim for 5-10 bullets.

### Step 3: List What Does NOT Change

Explicitly call out what's out of scope:
- "Database schema remains unchanged"
- "Existing REST API endpoints unmodified"
- "User authentication flow stays as-is"
- "No changes to deployment pipeline"

Aim for 3-5 bullets.

### Step 4: Identify Primary Decisions

Look for **blocking questions** that need answers before work can proceed:

**Template:**
```json
{
  "decision_id": "D1",
  "statement": "Which validation library to use?",
  "options": ["jsonschema", "pydantic", "custom"],
  "impact": "HIGH",
  "blocking": "YES"
}
```

**Impact levels:**
- HIGH: Affects multiple phases, changes architecture
- MEDIUM: Affects one phase, local scope
- LOW: Implementation detail, easily reversible

**Blocking:**
- YES: Cannot proceed without decision
- NO: Can defer decision to later phase

Only include decisions that are:
- Actually uncertain (not already decided)
- Need stakeholder input
- Block progress if not resolved

If no decisions needed, explicitly state: `[]` (empty array)

### Step 5: Identify Top 3 Risks

Use the risk register from SEC-013 but summarize the TOP 3 here:

**Template:**
```json
{
  "risk_id": "RISK-001",
  "type": "risk",
  "description": "External API might rate-limit requests",
  "impact": "MEDIUM",
  "probability": "HIGH",
  "mitigation": "Implement exponential backoff and request caching"
}
```

**Risk categories:**
- Technical: Dependencies, integration, performance
- Process: Timeline, resource availability
- External: Third-party APIs, vendor changes
- Resource: Skills, tools, budget

### Step 6: Define High-Level "Done"

Write 5 criteria that must ALL be true for plan completion:

**Examples:**
- "All 47 test cases pass in CI"
- "Code coverage ≥ 90%"
- "Security scan shows 0 high/critical issues"
- "Documentation updated with API examples"
- "Deployed to staging and smoke tests pass"

Must be:
- Measurable (not subjective)
- Binary (yes/no, not "mostly")
- Verifiable by command/script

---

## Validation Rules

1. `outcome_one_sentence` MUST be ≤ 150 characters
2. `what_changes` MUST have 5-10 items
3. `what_does_not_change` MUST have 3-5 items
4. Each decision MUST have: statement, options (≥2), impact, blocking
5. MUST have exactly 3 risks (or mark as RISK-NONE-001 if truly no risks)
6. `done_at_highest_level` MUST have exactly 5 criteria

## Common Mistakes

1. **Outcome too long** - Keep ≤ 150 chars, one sentence only
2. **Vague changes** - "Improve performance" not specific enough
3. **Fake decisions** - Don't list already-decided items
4. **Weak risk mitigation** - "Hope it doesn't happen" is not mitigation
5. **Subjective done criteria** - "Good test coverage" is not measurable
