# SEC-001: Plan Identity Instructions

**Schema File:** `sections/sec_001_plan_identity.json`  
**Section ID:** `sec_001_plan_identity`  
**Required:** Yes  
**Updated:** 2026-01-28

---

## What To Do

Fill in all plan metadata, ownership information, timestamps, scope, and fixed assumptions based on the user request.

---

## How To Do It

### Step 1: Generate `plan_id`

**Format:** `PLAN-{PROJECT_CODE}-{YYYYMMDD}-{NNN}`

1. Extract PROJECT_CODE from user request:
   - 2-15 characters
   - Uppercase letters, numbers, hyphens only
   - Example: "JSON validator" → "JSON-VALIDATOR"
   - Example: "Build API service" → "API-SERVICE"

2. Use current date (YYYYMMDD format): `20260128`

3. Start sequence number at 001 for first plan of the day

**Example:** `PLAN-JSON-VALIDATOR-20260128-001`

### Step 2: Extract `plan_title`

- Use first meaningful sentence from user request
- Remove filler words: "please", "can you", "I need", "help me"
- Maximum 100 characters
- Capitalize first letter only

**Examples:**
- Input: "Can you please build a CLI tool for JSON schema validation?"
- Output: "Build CLI tool for JSON schema validation"

### Step 3: Set `plan_version`

- New plans: `0.1.0`
- Revisions: increment patch version (0.1.1, 0.1.2, etc.)
- Major changes: increment minor version (0.2.0)

### Step 4: Set `plan_status`

Options: `Draft`, `Review`, `Approved`, `In Progress`, `Blocked`, `Done`

- Always start with: `Draft`
- Change when explicitly requested

### Step 5: Set `owner_dri`

- Use requesting user's email/username
- If not provided: `UNASSIGNED`
- DRI = Directly Responsible Individual

### Step 6: Set `reviewers`

- Leave empty `[]` unless explicitly mentioned
- If mentioned: `["user1@example.com", "user2@example.com"]`

### Step 7: Set Timestamps

- `created_date`: Current date (YYYY-MM-DD)
- `last_updated_date`: Current date (YYYY-MM-DD)
- `last_synced_at`: Leave as `null`

### Step 8: Determine `target_environments`

Infer from request or default to: `["windows", "linux", "macos"]`

Specific hints:
- "Windows only" → `["windows"]`
- "Docker" → `["docker"]`
- "AWS" / "cloud" → `["cloud"]`

### Step 9: Determine `execution_surface`

Options: `repo`, `service`, `library`, `infra`, `docs`, `mixed`

- Code files being modified → `repo`
- Deploying a service → `service`
- Building reusable component → `library`
- Infrastructure/config only → `infra`
- Documentation only → `docs`
- Multiple categories → `mixed`

### Step 10: Set `repo_or_workspace_root`

- Absolute path or URL
- If not provided, use: `{{PATH_OR_URL}}`
- Example: `C:\Users\richg\Gov_Reg`
- Example: `https://github.com/org/repo`

### Step 11: Set `primary_branch`

Default: `main`

Other options: `master`, `develop`

### Step 12: Add `related_documentation`

- Leave empty `[]` unless user references docs
- If mentioned: add path and type (spec|design|rfc|runbook)

### Step 13: Keep Fixed Assumptions

**DO NOT CHANGE** unless explicitly overridden:
- `team_model`: `solo_developer`
- `time_constraints`: `none`

---

## Validation Rules

1. `plan_id` MUST match pattern: `^PLAN-[A-Z0-9-]{2,15}-\d{8}-\d{3}$`
2. All dates MUST be YYYY-MM-DD format
3. `plan_version` MUST follow semantic versioning (X.Y.Z)
4. `repo_or_workspace_root` MUST be absolute path or valid URL
5. `primary_branch` MUST be one of: main, master, develop
6. `execution_surface` MUST be one of: repo, service, library, infra, docs, mixed

## Common Mistakes

1. **Using relative paths** - Always use absolute paths
2. **Forgetting to uppercase PROJECT_CODE** - Must be uppercase
3. **Including filler words in title** - Remove "please", "can you", etc.
4. **Changing fixed assumptions** - Leave team_model and time_constraints alone
5. **Invalid plan_id format** - Must follow exact pattern
