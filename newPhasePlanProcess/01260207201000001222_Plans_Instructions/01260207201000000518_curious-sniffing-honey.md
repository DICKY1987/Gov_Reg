#### 1. FM Script Schema Mismatch Analysis

**Status: ALL 12 FM scripts are incompatible with v3.0.0 schema**

The FM scripts expect a different schema structure:

**What FM scripts expect:**
- `plan.artifacts` as keyed dict with artifact records containing: `consumer_phases`, `producer_phase`, `schema_ref`, `location`
- `plan.phases_by_id` as dict (not array) with fields: `depends_on`, `validation_gates`
- `plan.gates_by_id` as dict (not array) with fields: `evidence_output`, `validation_command`, `success_pattern`
- `plan.commands` as flat dict with fields: `retry_policy`, `idempotent`, `command`

**What v3.0.0 schema provides:**
- `plan.artifacts` is config-only (registry type, paths, evidence_root) - NOT artifact records
- `plan.phases` as array
- `plan.gates` as array
- `plan.commands` as nested object with `standard` and `custom` properties

**Partial mitigation discovered:**
The CLI's `normalize_plan()` function (lines 172-186) already converts `phases` array → `phases_by_id` dict and `gates` array → `gates_by_id` dict. However, it doesn't address the `artifacts` structure issue.

- CLI code references `01260207233100000674_NEWPHASEPLANPROCESS_plan.schema.v2.4.0.json` but file not found

- **v3.0.0 is a "hybrid"** per schema line 4: "Hybrid schema combining draft step-level contracts with v2.4.0 operational features"
- Confirms `phases` and `gates` are **arrays** (lines 111-124)
- Confirms `artifacts` is **config-only** (line 89-90 references `$defs/artifacts_config`)

### Root Cause Clarification
The issue isn't "schema version mismatch" - it's **missing v2.4.0 schema file entirely**. The FM scripts and CLI were written for v2.4.0, but only v3.0.0 schema exists in the repo. The CLI's normalize_plan() was designed to convert v2.4.0 arrays to dictionaries, but there's no v2.4.0 schema to validate against.

## Phase 4: Final Plan

## Context

The newPhasePlanProcess system is a deterministic plan validation system with 19 required gates and 12 Failure Mode (FM) checks. The system has evolved through two schema versions:

- **v2.4.0**: Uses dictionary-based structures (`phases_by_id`, `gates_by_id`)
- **v3.0.0**: Uses array-based structures (`phases`, `gates`) with step-level contracts

The CLI includes a `normalize_plan()` function that converts v3.0.0 arrays into v2.4.0 dictionaries, enabling backward compatibility.

**Critical Issue:** All 12 FM scripts (FM-01 through FM-12) expect a `plan.artifacts` dictionary containing artifact records with fields like `consumer_phases`, `producer_phase`, `schema_ref`, and `location`. This structure doesn't exist in either v2.4.0 or v3.0.0 schemas, causing all FM checks to fail.

This plan addresses 4 interconnected tasks to make the system fully operational:

1. **Fix FM script schema mismatch** - Enable FM checks to run successfully
2. **Create comprehensive test plan** - Validate the entire gate pipeline
3. **Generate missing documentation** - Complete FM_GATE_MAPPING_CANONICAL.md and TECHNICAL_DOCUMENTATION.md
4. **Component analysis** - Deep dive into critical components as needed
