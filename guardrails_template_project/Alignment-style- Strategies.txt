It sounds like you are trying to build a **Style Guide** or a **Schema** for your CLI app. When you don't provide a "source of truth," the AI falls back on its training data, which might favor one style (like functional programming) while your existing scripts favor another (like object-oriented).

To stop the AI from guessing, you need to define **Constraints** and **Conventions**. Here are the primary categories you should define to ensure all generated code aligns perfectly with your existing ecosystem:

---

## 1. Programming Paradigm
This is the "soul" of your code. If you don't pick one, the AI will mix them, creating a "Frankenstein" codebase.
* **Procedural:** Linear, function-based scripts (Best for simple automation).
* **Object-Oriented (OOP):** Class-based, using inheritance and encapsulation (Best for complex tools).
* **Functional:** Emphasizes immutability and pure functions (Best for data processing).

## 2. Error Handling Philosophy
How should the app fail?
* **Look Before You Leap (LBYL):** Checking if a file exists before opening it (`if os.path.exists...`).
* **Easier to Ask Forgiveness (EAFP):** The standard Pythonic way. Using `try/except` blocks to catch errors as they happen.
* **Silent vs. Verbose:** Should the app log the error to a file, or print a full Traceback to the user?

## 3. Dependency Preferences
AI often suggests heavy libraries when a built-in one would work.
* **Standard Library Only:** Forces the AI to use `json`, `urllib`, and `pathlib` (No `pip install` required).
* **Third-Party Specifics:** Explicitly tell it: *"Use `rich` for CLI formatting and `httpx` instead of `requests`."*

## 4. Documentation & Type Hinting
This ensures the code is maintainable.
* **Type Hinting:** Will you use PEP 484 type hints (e.g., `def name(id: int) -> str:`)?
* **Docstring Format:** Google Style, NumPy Style, or Sphinx?
* **Comment Density:** Should it explain "what" the code does or "why" it does it?

## 5. Interface & User Experience (CLI Specific)
* **Argument Parsing:** Will you use `argparse` (built-in), `Click` (polished), or `Typer` (modern/type-hint based)?
* **Output Format:** Should the script output plain text, JSON (for piping to other tools), or "Pretty" tables?

## 1. Naming Conventions (Casing)
AI often flips between styles based on the libraries it sees. You must lock this down.
* **Variable/Function Names:** `snake_case` (Python standard) vs. `camelCase`.
* **Constants:** `UPPER_SNAKE_CASE` for hardcoded values.
* **Private Members:** Should internal functions start with an underscore (`_internal_func`) to signal they shouldn't be called externally?

## 2. File & Path Handling
This is a major source of "clash" between Windows and Mac/Linux users, or older vs. newer Python versions.
* **Pathlib vs. Os:** Explicitly require `from pathlib import Path`. This treats paths as objects rather than strings, preventing slash errors (`/` vs `\`).
* **Relative vs. Absolute:** Define if the app should always look for config files relative to the **user's Home directory**, the **Current Working Directory (CWD)**, or the **Script Location**.

## 3. Data Serialization Preference
When your app saves or moves data, how does it do it?
* **Format:** `JSON` (standard), `YAML` (human-readable config), or `TOML` (the new Python standard for configuration).
* **Schema Validation:** Do you want the AI to use `Pydantic` to enforce data types, or just raw dictionaries?

## 4. Logging vs. Printing
In a CLI app, "Printing" is for the user; "Logging" is for the developer.
* **The UI:** Should the app use `print()` for everything, or a library like `Rich` to create beautiful layouts and progress bars?
* **The Audit Trail:** Should the app create a `.log` file? If so, what level of detail? (`DEBUG`, `INFO`, `ERROR`).

## 5. Concurrency Model
If your app needs to do two things at once (like downloading a file while showing a spinner):
* **Synchronous:** One task at a time (e.g., `requests`).
* **Asynchronous:** Non-blocking tasks (e.g., `asyncio` and `httpx`).
* **Multiprocessing:** Using multiple CPU cores for heavy math/data processing.
