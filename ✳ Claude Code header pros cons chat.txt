  ## 1) Scope and intent

  ### Objective

  Refactor the registry system to achieve:

  * **Deterministic automation** (contract-driven population, fail-closed gates).
  * **High traceability** (explicit provenance/evidence per derived value and per relationship).
  * **Low sparsity / low “weirdness”** by separating canonical stores by record kind while preserving “one SSOT” semantics via a **bundle
  manifest**.

  ### In-scope changes

  1. **Enum canon + drift gate** (preflight hard stop).
  2. **Canonicalize the entity set** (stop having 3 parallel “entity-ish” arrays without governance).
  3. **Backfill** the schema-required FileRecord fields so the canonical store is actually schema-valid at scale.
  4. **Edge generation from evidence** (not “extracting existing relationships,” because they’re empty today).
  5. **Generators registry** (tool/run records become first-class).
  6. **Split canonical registries + bundle manifest + derived unified view**.

  ### Non-goals (explicit)

  * Not solving “perfect classification” in one pass. This design mandates **repeatable, evidence-backed inference**, not omniscience.
  * Not replacing your contract model (Column Dictionary + Derivations + Write Policy). This design tightens and extends it.

  * `REGISTRY_file.json` contains 4 arrays: `files` (2013), `entities` (723 stubs), `entries` (245 legacy overlay), `edges` (0).

    * `artifact_kind`, `layer`, `governance_domain` populated for only **34/2013**.

  What's strong

  Phasing is correct and disciplined. Each phase has a clear "why before next" dependency. The ordering prevents the classic mistake of reorganizing

  What I'd tighten

  1. §7.3 merge precedence needs a conflict resolution spec

  You define precedence order (canonical > overlay entries > stubs) but not what happens on field-level conflict. Example: if files[] has
  governance_domain: null and entries[] has a GEU role that implies a governance domain, does the overlay win for that field? You need a per-field
  or per-field-category rule, not just a record-level precedence.

  Recommend adding a small table: "overlay-writable fields" (the GEU/bundle-role fields from entries[]) vs "canonical-only fields" (everything
  else). That makes the merge deterministic without ambiguity.

  2. §8.1 backfill needs explicit "unknown/unclassified" handling

  The backfill algorithm says "infer artifact_kind, layer, governance_domain" but doesn't specify what happens when inference fails. For 2,013
  records with 98% null, some will be unclassifiable (e.g., files with no extension, no convention match, no content signal).

  Options:
  - Add UNCLASSIFIED to the enum canon as a valid temporary value
  - Leave null and accept <100% schema validity
  - Reject the record from the canonical store

  You should pick one explicitly. I'd lean toward UNCLASSIFIED as a canonical enum value with a contract rule that it triggers re-evaluation on next
   scan — otherwise your 95% gate threshold is aspirational with no fallback.

  3. §9.2 edge discovery ordering should also specify what ships in Phase 2 vs. later

  You list three tiers (static parsing, conventions, heuristics) but don't scope which are Phase 2 deliverables vs. future. For a registry with
  2,013 entities, static import parsing alone could produce a meaningful edge set. I'd recommend:

  - Phase 2a: static parsing only (Python imports, JSON/YAML refs, explicit config references)
  - Phase 2b: convention-based pairing (test↔code, validator↔schema)
  - Phase 2c (optional/future): heuristic inference

  This prevents Phase 2 from becoming unbounded.

  4. §10.3 generator↔entity linkage has a schema gap

  You note "if your edge schema is file-centric, you can implement generator linkage as generator→file edges... or extend EdgeRecord target types in
   schema v4." This needs to be a decision, not an option. Since you're defining schema v4 anyway, I'd recommend extending EdgeRecord to support
  target_generator_id as an alternative to target_file_id / target_schema_id. Otherwise generators become second-class citizens in the edge graph.

  5. §11.2 atomic commit needs a rollback spec

  The protocol says "archive previous bundle before each write" but doesn't specify what happens on partial failure. If entities validates, edges
  validates, but generators fails schema — do you roll back the whole bundle or commit the valid parts? Given your invariant that "a commit is valid
   only if all artifacts validate," the answer should be full rollback, but state it explicitly and specify where the failed bundle goes (staging
