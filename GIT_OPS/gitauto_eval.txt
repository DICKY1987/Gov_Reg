gitauto_eval

Active evaluation for the GitHub-native mapping of `newPhasePlanProcess`.

The core mapping is sound:

- phases should live in a Project `Phase` field, not in milestones
- milestones should represent delivery checkpoints
- one step should map to one issue by default
- gates should be required status checks
- pull requests should be the execution attempts linked to governed issues

The remaining work is in implementation discipline, not in the basic mapping.

---

## Active cautions

### 1. Prefer GitHub App auth over PAT-based project automation

Personal access tokens are workable for prototypes, but they are the wrong
default for repeatable project automation. Use GitHub App installation tokens
for project mutations whenever possible.

### 2. Do not hardcode project field IDs or option IDs

Project, field, option, and iteration IDs are environment-specific. Any workflow
that writes project fields should resolve them through a bootstrap step or a
generated registry rather than embedding IDs inline.

### 3. Keep phase advancement criteria explicit

Built-in project workflows are useful for simple status moves, but they do not
replace explicit phase-completion logic. If a phase transition depends on more
than one condition, define the conditions clearly and evaluate them in one place.

### 4. Separate platform-agnostic workflows from Windows-dependent work

GitHub-hosted Linux runners are fine for GraphQL, REST, field bootstrap, and
light project automation. Any workflow that depends on Windows-specific tools,
PowerShell behavior, or local execution assumptions should be isolated and
routed deliberately.

### 5. Treat preview-only GitHub features as optional

Organization-wide issue fields are useful, but they should not become a hard
dependency until they are stable in your target environment. Project custom
fields are the safer default baseline.

### 6. Make the NPP-to-GitHub state mapping explicit

If `newPhasePlanProcess` still informs execution behavior, define the mapping
from NPP concepts to GitHub states directly:

- phase -> project phase field
- step -> issue
- gate -> required check
- execution -> pull request
- completion -> issue closure plus milestone/project progress

Do not leave that bridge implied across multiple files.

### 7. Document field ownership before adding more automation

When workflows, humans, and built-in automations can all touch the same field,
silent conflicts become inevitable. Decide which fields are human-managed,
workflow-managed, or built-in-managed before expanding automation.
