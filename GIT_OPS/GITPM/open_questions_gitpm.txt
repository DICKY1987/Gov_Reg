# Active Open Questions for GitHub-Native Planning

This file keeps only the questions that still apply under the current GitHub-native-first direction.

The original mixed document, including the archived custom control-plane questions, is preserved at `GIT_ARC/GITPM/open_questions_gitpm_original.txt`.

## Active Questions

**OQ-06: What are the stuck-item thresholds for backlog hygiene?**

**Recommendation:** Define a versioned hygiene policy with per-status max age in days and a matching escalation action. Example: `Backlog=14`, `In Progress=5`, `In Review=3`, `Blocked=2`. Keep the policy in repo so the workflow logic stays simple and the thresholds can change without rewriting automation.

---

**OQ-13: How are `APP_ID`, `APP_PEM`, `ORGANIZATION`, and `PROJECT_NUMBER` provisioned and validated?**

**Recommendation:** Add a short setup checklist and a manual validation workflow. The validation workflow should confirm that the GitHub App token can be minted and that the target project is reachable before any operational workflow runs.

---

**OQ-14: Which fields are automation-managed versus human-managed?**

**Recommendation:** Define a field ownership matrix. At minimum, document whether `Phase`, `Status`, milestone/date fields, assignees, labels, and any gate/evidence fields are automation-managed, human-managed, or shared. GitHub-native does not remove the need for single-writer rules.

---

**OQ-16: What saved project views are required?**

**Recommendation:** Define a minimum standard view set for every governed project. Recommended baseline: `Backlog`, `Execution Board`, `Roadmap`, `Blocked`, and `Phase Review`. Treat these views as part of the template, not as optional local customization.

---

**OQ-17: How is the project template governed and versioned?**

**Recommendation:** Choose one canonical project template and track its version explicitly. Document which fields, views, workflows, and hierarchy features are required, and add a periodic validation pass so active projects do not quietly drift away from the template.

---

**OQ-18: What is the draft-item intake policy before work becomes an execution issue?**

**Recommendation:** Use draft items for raw intake, convert them to issues only after qualification, and apply execution fields only after promotion. This keeps ideation separate from tracked delivery work and avoids polluting the execution board with unqualified items.

---

**OQ-19: How are native hierarchy features enabled and verified?**

**Recommendation:** Make `Parent issue` and `Sub-issue progress` explicit prerequisites. Document the required org-level settings, the manual project configuration steps that cannot be bootstrapped by API, and a validation check that fails clearly when hierarchy support is missing.

---

**OQ-20: What is the project status update policy?**

**Recommendation:** Decide who drafts, reviews, and publishes project-level status updates. A practical default is automation-generated inputs with human-reviewed published updates, so executive-facing status stays accurate without becoming a manual reporting burden.

---

**OQ-21: What minimum insights and charts pack is required?**

**Recommendation:** Define a standard reporting set. Recommended baseline: items by phase, items by status, milestone progress, blocked-item count, and stale-item count. Use native GitHub insights where they exist and keep any custom reporting lightweight.

---

**OQ-22: Where should planning metadata live: project fields or issue fields?**

**Recommendation:** Prefer project fields for process-governance metadata and use issue fields only when there is a clear human-driven need that benefits from issue-level visibility. Document the exception cases explicitly so teams do not split the same concept across two surfaces.

---

**OQ-23: What is the response to unauthorized UI edits on automation-managed fields?**

**Recommendation:** Define the operator response even if enforcement stays lightweight. At minimum, document how such edits are detected, who reviews them, and whether the expected action is manual revert, comment-based warning, or a scheduled validation workflow.
