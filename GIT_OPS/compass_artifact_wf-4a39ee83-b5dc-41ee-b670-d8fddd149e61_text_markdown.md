# SSOT-to-Execution-Surface Projection: A Cross-System Architecture Analysis

**Every mature system that separates planning from execution converges on the same primitive: a reconciliation loop that continuously diffs desired state against observed state and acts to close the gap.** This pattern appears in workflow engines (Temporal), infrastructure delivery (ArgoCD/Flux), database schema management (PlanetScale/Atlas), and — imperfectly — in project management tooling (Linear, GitHub Projects v2). The critical finding across all seven systems studied is that the "projection compiler" pattern the query describes does not yet exist as a named, formalized architecture for project management, though all the constituent primitives are proven in adjacent domains. What typically breaks is not the projection itself but the reconciliation: bidirectional sync conflicts, impedance mismatch between rich SSOT schemas and impoverished execution surfaces, and the absence of escape hatches for intentional manual overrides.

---

## The reconciliation loop as universal primitive

The Kubernetes controller pattern — **observe → diff → act → report** — is the architectural bedrock underlying every system in this analysis. Temporal replays workflow definitions against event histories. ArgoCD compares rendered Git manifests against live cluster objects. PlanetScale diffs branch schemas via Vitess's schemadiff library. Atlas introspects live databases and generates corrective DDL. In every case, the system maintains two representations of reality and continuously works to align them.

The key architectural fork is between **point-in-time reconciliation** (Terraform, Pulumi, PlanetScale deploy requests) and **continuous reconciliation** (Kubernetes operators, ArgoCD, Flux, Crossplane, SchemaHero). Point-in-time systems run when commanded — `terraform apply`, a deploy request approval — and drift accumulates between runs. Continuous systems loop indefinitely, typically on **3–10 minute intervals**, treating convergence as an ongoing process rather than a discrete event. ArgoCD defaults to 3-minute polling with optional webhook triggers for immediate response. Flux uses per-resource intervals with separate `retryInterval` for faster failure recovery.

A crucial design principle separates robust systems from fragile ones: reconciliation must be **level-triggered, not edge-triggered**. Both ArgoCD and Flux care about *current state*, not *what events occurred*. If a webhook is missed, the next polling cycle catches the drift. Temporal's deterministic replay works the same way — it regenerates the expected command sequence from scratch on every replay rather than depending on a log of transitions. This makes all three systems self-healing by construction. Systems that rely on edge-triggered event processing (single GitHub webhook → update project field) are inherently more fragile because missed events create permanent state divergence.

## How each system separates definition from execution

**Temporal.io** provides the cleanest separation. The Workflow Definition — deterministic code written in Go, Java, TypeScript, or Python — serves as the SSOT for "what should happen." The Event History — an immutable, append-only log of every action, result, and state transition — represents what actually happened. During replay, Workers re-execute the Workflow Definition from the beginning, comparing generated Commands against recorded Events position by position. If a `ScheduleActivityTask` command doesn't match an `ActivityTaskScheduled` event at the same position, a **non-determinism error** fires. Crucially, this error does *not* fail the workflow — it creates a Workflow Task Failure that retries indefinitely, preventing incorrect forward progress while giving operators time to deploy a fix.

Temporal handles definition-state drift through two versioning strategies. **Patching** (the `patched()` function) inserts Marker Events into the history, creating logical branches where replay follows old code paths for existing executions and new paths for new ones. **Worker Versioning** routes executions to specific Worker deployments, avoiding code branching entirely. The recommended defense is **replay testing in CI/CD**: take production histories, replay against new code, verify no non-determinism errors before deployment. All state updates — event history, task queues, timers, mutable state — are written **transactionally** within a shard. As Maxim Fateev stated: "If you run atop a workflow engine that doesn't have transactions across all these components, you will run into all sorts of race conditions."

**GitOps (ArgoCD/Flux)** implements a layered reconciliation that extends Kubernetes' native desired/observed state model to an external SSOT. ArgoCD stores desired state as an `Application` CRD's `.spec.source` (Git repo, path, revision) and maintains observed state in a live cluster cache backed by Redis. Sync status lives in `Application.status.sync` (Synced/OutOfSync/Unknown) alongside health status (Healthy/Progressing/Degraded/Suspended/Missing/Unknown). Flux decomposes this further through specialized CRDs: `GitRepository` for source tracking, `Kustomization` for manifest application, and `HelmRelease` for chart management — each with independent reconciliation loops communicating through artifact references.

ArgoCD's drift detection has evolved from client-side manifest diffing to **server-side dry-run apply**, which eliminates false positives from API server defaulting and mutation webhooks. The system provides extensive `ignoreDifferences` configuration via JSON pointers, JQ expressions, and managed-fields-manager exclusions for handling fields mutated by other controllers (HPA scaling, Istio sidecar injection, cert-manager certificates). Flux natively uses Kubernetes server-side apply, leveraging the `flux-client-side-apply` field manager for ownership tracking.

**PlanetScale** applies the same pattern to database schemas. Development branches create isolated schema copies. All changes flow through **deploy requests** — analogous to pull requests — where Vitess's schemadiff library generates semantic diffs. A **three-way merge** algorithm handles concurrent changes: identical changes are allowed, incompatible changes are blocked, and unrelated changes proceed independently. Safe migrations optionally **block direct DDL entirely**, converting the preventive-vs-detective tradeoff into a purely preventive one. This is the strongest enforcement pattern found across all systems studied: rather than detecting unauthorized changes after the fact, the system makes them impossible.

**Linear.app** separates planning concerns orthogonally. **Projects** track delivery (what you're building), **Cycles** track iteration (when you're building it), and **Issues** are the atomic SSOT — every other grouping derives its state from issue statuses. An issue simultaneously belongs to one project and one cycle, allowing delivery tracking and sprint planning to operate independently. **Initiatives** sit above projects for strategic alignment. The GitHub integration operates in two modes: PR/branch linking (unidirectional, rich — PR status drives Linear issue status) and GitHub Issues Sync (bidirectional, lossy — only title, description, status, labels, assignee, and comments sync). Status mapping is fundamentally impoverished: Linear's **6+ workflow states across 5 categories** compress into GitHub's binary open/closed model.

## The projection compiler gap revealed by Oxide and GitHub Projects

The most striking finding of this research is that **no established pattern, RFC, or widely-adopted tool exists for what the query describes as a "projection compiler"** — a system that takes a declarative spec and compiles it into mutable project management artifacts (issues, milestones, boards) with ongoing reconciliation. The term does not appear in any published architecture document. The closest analogues exist in adjacent domains:

**Oxide Computer's RFD process** demonstrates a rigorous SSOT for engineering decisions — **over 500 RFDs comprising approximately 1.4 million words** stored as AsciiDoc in a private Git repository, with a six-state lifecycle (prediscussion → ideation → discussion → published → committed → abandoned). Oxide has built substantial tooling: an `rfd-api` backend in Rust that processes GitHub webhooks and stores revisions in Postgres, an `rfd-site` frontend with Meilisearch-powered full-text search and inline PR comment rendering, a chat bot for fuzzy-matching RFD titles, and short URLs (`{num}.rfd.oxide.computer`). But critically, **Oxide does not implement automated projection from RFDs to GitHub Issues**. The relationship is informal: RFDs define the "what and why," code repos reference relevant RFDs in documentation, and post-implementation feedback goes to GitHub Issues. There is no reconciliation loop between planning documents and execution artifacts.

**Red Hat's `qontract-reconcile`** is the closest real-world implementation of the projection compiler concept. It takes declarative definitions from a central "App-Interface" data store and reconciles dozens of external systems — GitHub teams, Terraform resources, Slack channels, DNS records — with `--dry-run` support analogous to `terraform plan`. **GitHub's Spec Kit** (released September 2024) formalizes spec-driven development for AI coding agents with a specs → plans → tasks → implementation workflow, but lacks a reconciliation loop. GitHub's recent **Agentic Workflows** compile Markdown descriptions into `.lock.yml` GitHub Actions files via `gh aw compile` — literally a "compile markdown to operational system" pattern, though for CI/CD rather than project management.

The Terraform GitHub Provider can declaratively manage repositories, teams, permissions, and branch protections with full state tracking and drift detection. But it **cannot manage individual issues or project board items**. No "Terraform for Issues" exists. The tools that create issues from files (`peter-evans/create-issue-from-file`) are fire-and-forget with no reconciliation. This gap — between the mature infrastructure reconciliation patterns and the immature project management automation landscape — represents the core opportunity the query identifies.

## Production GitHub Projects v2 automation remains primitive

GitHub Projects v2 field IDs use typed prefixes (**`PVTF_`** for generic fields, **`PVTSSF_`** for single-select, **`PVTIF_`** for iteration fields) and are stable once created — they persist across field renames and only change if a field is deleted and recreated. A significant production issue was discovered where webhook events sent field IDs in `PVTF_` format while the GraphQL API returned `PVTSSF_` format for single-select fields; GitHub has since shipped a fix to align these. The standard bootstrap pattern, recommended in GitHub's own documentation, is a **query-then-extract** approach: fetch all fields via GraphQL at workflow start, extract IDs by name using `jq`, and cache them as environment variables.

For authentication at scale, GitHub explicitly recommends **GitHub Apps for organization projects** and PATs only for user projects. The `GITHUB_TOKEN` in Actions cannot access projects at all. GitHub App installation tokens have a **1-hour maximum lifetime** and scale from 5,000 to 12,500 requests/hour based on organization size (plus 15,000/hr for GHEC), versus a fixed 5,000/hr for PATs. The `actions/create-github-app-token@v2` action is the production standard for minting installation tokens. Organizations with SAML SSO must explicitly authorize PATs — a common gotcha documented across multiple automation repositories.

Built-in Projects workflows support only **single-event triggers** (item added → set status, PR merged → set Done). There is no native composite condition evaluation. The community has actively requested this (Discussion #53973). Production teams implement composite evaluation through three patterns: GitHub Actions workflows with multi-event triggers and conditional `if` guards, external webhook receivers (AWS Lambda, custom servers) that query full item state and evaluate conditions before mutating, and **scheduled reconciliation workflows** (`cron: '*/15 * * * *'`) that periodically sweep all items and apply composite logic. The IssueOps pattern from GitHub's blog provides the conceptual framework — states, events, guards, and transitions — but requires custom implementation. No purpose-built drift detection tools exist for GitHub Projects v2. The June 2024 webhook enhancement that includes **previous and current field values** in `projects_v2_item` payloads enables real-time drift detection but requires custom infrastructure to act on.

No public production-grade automation architectures were found from Shopify, HashiCorp, or Stripe for GitHub Projects v2 despite extensive searching. The most comprehensive open-source action is `leonsteinhaeuser/project-beta-automations`, which handles name-based field resolution, both PAT and GitHub App auth, and custom field management — but it operates on single-event triggers, not composite conditions.

## What breaks in SSOT-to-execution-surface mapping

Across all seven systems, failure modes cluster into five categories.

**Impedance mismatch** is the most fundamental problem. Linear's rich workflow states (In Review, Ready to Merge, Won't Fix, Could Not Reproduce) compress into GitHub's binary open/closed. Temporal's deterministic workflow definitions cannot express inherently non-deterministic operations (I/O, randomness, system clocks) without delegating to Activities. Declarative schema tools cannot natively express **renames** — an inherently imperative operation requiring add-column, copy-data, drop-column sequences. ArgoCD's continuous reconciliation model breaks traditional database migration tools that assume single CI pipeline execution. The execution surface invariably has a different "shape" than the SSOT, and the projection is always lossy.

**Bidirectional sync conflicts** corrupt data reliably. Linear uses last-write-wins with no merge conflict UI — simultaneous edits on both sides can silently overwrite changes. As practitioner Marcel Krčah documented: "In past projects, bi-directional sync was a cause of data loss, unresolved conflicts, and frustration. In one project, the company ended up with two employees manually copying data." Proven resolution strategies include CRDTs (mathematically guaranteed convergence, used by Redis Active-Active), field-level ownership (different systems own different fields), and manual review queues for ambiguous conflicts. The strongest pattern is to make the sync **unidirectional** — Linear → GitHub for status, GitHub → Linear for code activity — rather than attempting true bidirectional mirroring.

**Reconciliation storms** emerge when reconciliation logic is not idempotent or when it interacts with other controllers. ArgoCD reconciling resources that Istio sidecars, cert-manager, or HPAs legitimately modify creates infinite correction loops. ApplicationSet status field ordering changes from Maps can trigger continuous reconciliation cycles. The solution requires **exclusion lists** for intentionally dynamic fields: `ignoreDifferences` in ArgoCD, `lifecycle { ignore_changes }` in Terraform, field manager ownership in Flux. The anti-pattern is suppressing drift warnings via blanket ignore rules instead of fixing root causes — this creates blind spots where real drift goes unnoticed.

**Missing escape hatches** make pure SSOT enforcement dangerous during incidents. Intuit, operating **45 ArgoCD instances across 200 clusters with 20,000 applications**, explicitly advocates "be strategic, be pragmatic — GitOps most of the time" and intentionally deviates during emergencies. PlanetScale's safe migrations can be temporarily disabled for emergency DDL. Terraform's state can be manually modified via `terraform state` commands. Systems that lack escape hatches force operators to choose between following process and resolving incidents, creating pressure to abandon the SSOT model entirely during crises.

**State explosion from partial failures** creates the hardest debugging problems. In sharded database environments, schema changes can succeed on shard A but fail on shard B, leaving migration-based tools in an inconsistent version state. Terraform can apply a change but fail to write the state file, creating phantom drift. Temporal's event history has a **hard limit of 50,000 events or 50MB** — exceeding it terminates the workflow. The architectural response is either declarative convergence (Atlas/SchemaHero simply re-diff and re-converge regardless of intermediate failures) or Continue-As-New patterns (Temporal resets event history while preserving logical continuity).

## Proven idempotency patterns for projection engines

Five patterns consistently produce reliable projection engines across all domains studied.

**Transactional outbox** solves the distributed write problem. Temporal writes events, state updates, task queue entries, and timer updates in a single database transaction per shard. This eliminates the Two Generals Problem for local operations — either all state is persisted or none is. For cross-system projection (SSOT → GitHub), the pattern translates to: write the intended change to a local outbox table, then asynchronously project to the execution surface, with at-least-once delivery and idempotent consumers.

**Checksum-based change detection** prevents unnecessary mutations. Flyway uses file checksums for repeatable migrations. Atlas uses schema fingerprints. ArgoCD computes manifest hashes. The pattern for a projection engine: hash the SSOT specification, hash the current execution surface state, compare. If checksums match, skip. This eliminates reconciliation storms from benign re-evaluations and reduces API rate limit consumption — critical for GitHub's **5-point cost per GraphQL mutation** versus 1 point per query.

**Optimistic concurrency control** prevents concurrent reconciliation from corrupting state. Kubernetes rejects updates with stale `resourceVersion`. Event-sourced projections use stream position tracking — record the last processed sequence number and skip events at or below it. For GitHub Projects, the pattern translates to: read item state with its current field values, evaluate conditions, mutate only if the pre-conditions still hold, and retry with fresh state on conflict.

**Natural idempotency through convergent operations** eliminates the need for deduplication infrastructure. Declarative tools like Atlas, SchemaHero, and Skeema always compute the diff from current state to desired state, regardless of how many times they run. The same approach applies to projection engines: rather than "create issue if not exists," use "ensure issue exists with these properties" — upsert semantics where the operation converges to the same result regardless of current state. `INSERT ON CONFLICT UPDATE` in SQL, `server-side apply` in Kubernetes, and `updateProjectV2ItemFieldValue` in GitHub's GraphQL API all support this pattern natively.

**Exponential backoff with jitter** on reconciliation failures prevents thundering herds and API abuse. Kubernetes controller-runtime provides this out of the box via its work queue. ArgoCD supports `timeout.reconciliation.jitter` to spread reconciliation across time. For GitHub Projects automation, scheduled workflows should include randomized delays to avoid synchronized API storms across repositories.

## Conclusion

The architectural pattern this research explores — declarative SSOT projected into mutable execution surfaces with reconciliation — is **mature in infrastructure (GitOps, Terraform, database schema management) but nascent in project management**. The universal primitive is the observe-diff-act loop, operating level-triggered on configurable intervals. Temporal adds the deepest insight: by making the workflow definition literally replayable against its event history, it achieves a form of projection where the current state is always derivable from the SSOT plus recorded inputs — true event sourcing applied to workflow orchestration.

The "projection compiler" for project management does not yet exist as a named pattern or production tool. The gap is real: Oxide's RFDs are a rigorous planning SSOT with no automated projection to execution artifacts. Linear's GitHub sync is lossy and bidirectional-conflicting. GitHub Projects v2 automation lacks composite condition evaluation and drift detection primitives. The closest real-world implementation is Red Hat's `qontract-reconcile`, which reconciles declarative definitions against dozens of operational systems.

Three design principles emerge as non-negotiable for any system attempting this pattern. First, **the projection must be unidirectional** with clear ownership boundaries — bidirectional sync between SSOT and execution surface reliably produces data corruption. Second, **the reconciliation loop must be idempotent and level-triggered**, not dependent on event delivery guarantees. Third, the system must provide **escape hatches for intentional divergence** alongside drift detection that distinguishes unauthorized changes from deliberate overrides. The strongest enforcement pattern found — PlanetScale's safe migrations blocking direct DDL entirely — works precisely because it eliminates the category of unauthorized changes rather than trying to detect and revert them after the fact.
