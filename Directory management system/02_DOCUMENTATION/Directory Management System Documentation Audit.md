# Directory Management System Documentation Audit
## ID_REGISTRY.json

### Purpose and context

 This JSON file is the authoritative registry of all file identities in the system. It serves as the single source of truth for allocated 16-digit IDs, tracking the global SCOPE and every file’s registered ID and metadata. It is maintained by the identity allocation system (e.g. registry_store.py) and used to prevent ID collisions[1].
### Key concepts and schema

 The file defines a JSON structure with a top-level schema_version and the current scope value (six-digit project code)[1]. It contains two main sections:
- counters: A map of counter streams for ID allocation. Each key represents a (NS, TYPE, SCOPE) combination and stores the last allocated sequence number (current and allocated) and any reserved IDs[2][3]. In the current file, keys use the format NS_TYPE_SCOPE (underscores) – for example "999_01_260119" corresponds to namespace 999, type 01 under scope 260119[2][4].
- allocations: An array of records for each allocated ID. Each record includes the 16-digit id, the file_path it’s assigned to, timestamps (allocated_at), allocator (allocated_by), status, and metadata[5]. Metadata may include type_code and ns_code (two- and three-digit codes derived from file type and directory) and flags for legacy IDs[6]. Notably, records imported from an older scope are marked with legacy_id: true along with a legacy_scope field capturing their original scope (260118)[6]. This indicates those IDs were allocated under a previous scope and carried over, avoiding reallocation.
### Critical configuration values

 The scope field at the top is set to "260119" in this registry (a 6-digit code)[1]. This is the collision domain ensuring all IDs in this file share the same project scope. The counter keys reveal the counter key format currently in use: "NS_TYPE_SCOPE" (with underscores) for tracking sequences[3]. For example, the entry "999_01_260119": { "current": 9, ... } shows that for namespace code 999 and type code 01 in scope 260119, 9 IDs have been allocated[2][3]. The use of underscores with NS first is a configuration choice that differs from the intended canonical format (see SEQ_ALLOCATOR_SPEC below). The registry also encodes type and namespace codes per file (e.g. type_code: "00", ns_code: "999" for an uncategorized file) in metadata[6]. These codes originate from the IDENTITY_CONFIG.yaml rules and determine how the 16-digit ID is constructed (TT=type, NNN=namespace).
### Versioning and authority

 The file declares a schema_version (e.g. "1.0") indicating the version of the registry schema in use[1]. As the central registry, this JSON is considered the authoritative source for allocated IDs under governance policy. Any changes (e.g. scope change, schema updates) must follow approval and version bump procedures. The presence of schema_version and the registry’s role align with the Schema Authority Policy – it is the single source of truth for identity records, and documentation or tools should not deviate from what is captured here[7][8].

## SEQ_ALLOCATOR_SPEC.md

### Purpose and context

 This document is a formal specification for the ID sequence allocator. It defines how the 16-digit ID’s sequence portion (SEQ) is generated in a monotonic, deterministic way across file types and namespaces. Essentially, it is the contract that the identity allocation implementation must follow to ensure unique ID creation. The context is the “NUM16 TYPE+NS+SEQ+SCOPE” profile – meaning IDs have a 16-digit structure composed of Type, Namespace, Sequence, Scope[22]. The spec aims to guarantee consistency in ID generation across tools.
### Key concepts and rules

 The spec first outlines the ID segment definitions[23]:
- TYPE (TT) – 2 digits derived from the file extension via a classification table (e.g. “20” for .py, “01” for .md).
- NS (NNN) – 3 digits derived from the file’s directory or category via namespace routing rules (e.g. “100” for docs, “999” for uncategorized)[24].
- SEQ (#####) – 5 digits allocated by the sequence counter; this is the portion the allocator manages, ensuring each new file gets the next number[25][23].
- SCOPE (######) – 6 digits fixed for the project or system scope (configured in the identity config)[23].
The core rule is: “SEQ = next integer in the counter stream keyed by (TYPE, NS, SCOPE)”[25]. The document emphasizes hard invariants for allocation[26]: uniqueness (no two files share an ID), monotonic increase per key, no renumbering or reuse of IDs, a reserved value (00000 not used), and capacity limits (1–99999 for sequence)[26]. If an allocation would exceed 99999, it must fail (ensuring ID format consistency)[27].
The spec describes the data stores involved: a counter store (COUNTER_STORE.json) that persists the last allocated sequence per key, and an optional allocation ledger (audit log) for each allocation event[28][29]. Locking is mandated during updates to avoid race conditions (a file lock around counter store updates)[30].
### Critical configuration values

 This spec ties directly into configuration: it designates IDENTITY_CONFIG.yaml and the counter store as sources of truth[31]. The scope value is a fixed constant provided by the identity config (e.g. in YAML) that is baked into every ID[23]. The counter key format is explicitly defined as "{SCOPE}:{NS}:{TYPE}" (scope-first, colon-delimited) as the canonical format for keys in the counter store[32]. For example, "260118:200:20" would represent scope 260118, namespace 200, type 20[33]. This format is in line with the ID’s segment order (TT NNN SSSSS scope) and is intended to be used in the storage and retrieval of counters. (The spec notes this in contrast to an older format with underscores – see SCOPE_DECISION below).
Other critical values come from the identity config: type_code mappings (extension to 2-digit code) and ns_code mappings (directory patterns to 3-digit code) are referenced via the config file[34][24]. The identity config also supplies the project scope value (currently 720066 as per config) and a regex for detecting existing IDs in filenames. All these are considered authoritative inputs. The spec explicitly requires that TYPE, NS, and SCOPE be validated to match their expected formats (e.g. two digits, three digits, six digits respectively) before allocation proceeds[35].
### Versioning and authority

 The document has front-matter indicating version: 1.0.0 and status: active[22]. It also lists “source_of_truth” references, namely the identity config and counter store JSON and schema[31], signaling that this spec aligns with those artifacts. This implies any change to allocation logic must update this spec and vice versa. The spec is effectively an authoritative contract (with a document ID DOC-ID-SEQ-ALLOC-001) that the implementation (code) must follow. Deviations in practice (like a different counter key format) would be considered a violation of this contract[33][36]. The spec ensures determinism: given the same config and file inputs, the allocated ID must be the same (this is also echoed as a principle in the unified analysis)[37]. In summary, SEQ_ALLOCATOR_SPEC.md formalizes the rules of ID generation to uphold the governance policies (monotonic counters, single scope, etc.) in a versioned, reviewable manner.
## UNIFIED_SSOT_REGISTRY_ANALYSIS.md

### Purpose and context

 This is a comprehensive analysis and design proposal for evolving the current file-only ID registry into a Unified Single Source of Truth (SSOT) registry. The document reviews the existing system’s capabilities and gaps, and outlines a multi-phase plan to extend the registry to cover new record types (edges, assets, transients, etc.)[38][39]. In context, it was written around Jan 20, 2026 (post Phase 1 rollout) to guide future development (Registry v2.x). It serves both as a gap analysis and a roadmap for registry version 2.1 and beyond.
### Key concepts and proposals

 The analysis introduces the concept of managing all identity records in one unified structure with a discriminator (record_kind). Currently, the system implicitly only has entity records of kind “file”[40]. The proposal is to accommodate:
- Multiple Entity Kinds: files (current), plus assets (reusable templates/configs), transients (runtime sessions or processes), external references, etc., distinguished by a new entity_kind field[41][42]. All would be record_kind: "entity" in the unified registry but tagged by entity_kind.
- Relationship Edges: a new record type record_kind: "edge" to capture relationships between entities (e.g. “File A documents File B” or “Output X generated from Input Y”). Each edge would have fields like edge_id, source_entity_id, target_entity_id, rel_type, confidence, and evidence metadata[43][44]. This addresses the current lack of traceability between files (no dependency or provenance tracking in v1).
- Generator Records: a new type record_kind: "generator" for documenting derivation processes (builds or transformations). These would track which tool or process produced certain artifacts, inputs/outputs, and ensure derivation transparency[38][45]. Generator records carry fields like generator_id, generator_name, declared dependencies, build hashes, etc., making it clear how an artifact was produced[46].
The document enumerates current system strengths vs gaps. Strengths include: a production-ready file identity system, deterministic ID allocation, audit logging, and validation framework[47]. Gaps identified: no relationship tracking (no edges), no confidence/evidence model, no asset registry, no transient ID lifecycle, no generator metadata, etc.[38][48]. Each gap aligns with a proposed addition (e.g., add edges to cover relationships, add confidence scores, add asset records)[49][50].
A phased approach is detailed:
- Phase 1 – Extend current registry minimally by adding record_kind (set to "entity" for all existing) and entity_kind (set to "file" for all existing records), plus basic file metadata fields (filename, extension, and extract doc_id as the 16-digit prefix)[51][52]. This phase is backward-compatible (the current system essentially, which was implemented as v2.0).
- Phase 2 – Introduce Edge tracking: start recording record_kind: "edge" entries for relationships while keeping entity records unchanged[53]. This includes implementing detectors for relationships and enforcing referential integrity (source/target IDs must exist)[54].
- Phase 3 – Introduce Asset and Transient entity kinds: add new entity_kind values and allow allocation of those (with their own ID formats or identifiers like asset codes)[55]. For example, an entity_kind: "asset" might have an asset_id and canonical path, etc.[56]. Transient records would track ephemeral run IDs with TTL, etc.
- Phase 4 – Add Generator records: track derivation processes and link outputs to inputs (this also implies capturing build tool version, hash of source registry, etc., as described)[57][45].
### Critical rules and schema definitions

 The analysis essentially defines a superset schema of 80 columns for the unified registry (version 2.1)[58]. It breaks them down into: 9 core fields (common to all records: record_id, record_kind, status, created_utc, ... etc.), 40+ entity fields, 12 edge fields, and 15+ generator fields[59][43]. For example, core fields include record_id (a new primary key separate from the file’s 16-digit doc_id) and status (extended to include states like running, failed for transients)[60]. Entity fields include everything currently in use (id, file_path as relative_path, type_code, ns_code) plus new ones like sha256 content hash, size_bytes, and mapping fields (module_id, process_id, role_code) for potential module/process tracking[61][62]. Edge fields and generator fields as listed above. The document also introduces record_kind as the discriminator and suggests promoting some metadata (type_code, ns_code) to top-level fields in the new schema[63].
A key concept is maintaining SSOT (Single Source of Truth): the unified registry would avoid split “Registry A/B/C/D” scenario by merging all into one file[64]. The analysis stresses that JSON Schema will govern this unified format and that all tools should operate on one registry to prevent divergence[65][7]. It also highlights referential integrity as a new requirement – edges must refer to valid entity IDs, etc., which implies stronger validation rules on allocation[66].
### Versioning and authoritative sources

 This analysis is forward-looking and doesn’t itself declare a version number for the registry, but it references Registry Spec v2.1 and JSON Schema changes. In fact, it cites that a schema version field (e.g. schema_version: "2.0") will be used for versioning the registry format[67]. The document calls out that the JSON Schema is the ultimate authority in case of any drift between docs and implementation (anticipating the Schema Authority Policy)[68]. It also implicitly follows semantic versioning: major changes like adding a new record_kind require a major version bump (this aligns with policy)[69][70].
The conclusion of the analysis recommends an incremental, non-breaking migration – essentially treating the current system as Phase 1 (v2.0) and gradually adding features to reach the unified vision[71]. Until those phases are enacted, the current registry remains the authoritative source. This document itself is not a formal spec but heavily informs the official Registry v2.1 spec and Quick Reference (which were created to document the changes once decided). In summary, UNIFIED_SSOT_REGISTRY_ANALYSIS.md provides the rationale and blueprint for the authoritative changes that are now reflected in the v2.1 schema and policy.
## SCHEMA_AUTHORITY_POLICY.md

### Purpose and context

 This document establishes the governance rules for schema changes and expansions in the Unified SSOT Registry. It is a formal policy (classification: GOVERNANCE_POLICY) that ensures all stakeholders follow a single process for modifying the registry’s schema or enumerations[72]. The context is to prevent the registry schema from fragmenting or drifting – a risk when multiple teams or tools might try to introduce changes. By codifying authority, versioning, and approval workflow, it keeps the registry as a stable SSOT.
### Key principles and rules

 The policy’s Executive Summary states one authoritative source of truth for the registry schema and strict governance to avoid drift or breaking changes[73]. The core principles include:
- JSON Schema is Authoritative: The JSON Schema file (e.g. unified_ssot_registry.schema.json in the repository) is the ultimate authority on the registry format[74]. Documentation or code must reflect the schema, not vice versa. In any conflict, the JSON Schema wins and docs need updating[8]. This ensures the schema (which can be validated automatically) remains the single source of truth. The policy even outlines an “Authority Hierarchy” – 1) JSON Schema, 2) DERIVATIONS.yaml (for computed field logic), 3) WRITE_POLICY.yaml (for ownership rules), 4) documentation – in that order[75]. Documentation is explicitly non-authoritative; it should describe the schema but never override it[8].
- Semantic Versioning: All schema changes must use semantic versioning (MAJOR.MINOR.PATCH) and bump the meta.version accordingly[76][77]. The policy defines what constitutes a major vs minor vs patch change. For example, MAJOR (X.0.0) for breaking changes like removing or renaming fields, changing types, removing enum values, or adding required fields[78]. MINOR (0.X.0) for backward-compatible additions like adding optional fields or new enum values (if non-breaking)[79]. PATCH for documentation-only or non-functional updates[80]. Any edit to the schema requires a version bump, even for minor description tweaks, to maintain traceability[77].
- Enum Governance: The policy differentiates Closed Enums vs Open Enums vs Conditional Enums[81][82]. Closed enums (e.g. record_kind, or internal policy fields like writable_by) require a schema version bump and full governance process to add or change values[83][84]. Open enums (e.g. entity_kind, rel_type, evidence_method – domains expected to grow) can have new values added in a backward-compatible way without a major version, though still likely a minor bump and documentation update[85]. Conditional enums (like status, where some values apply only to certain kinds of records) require careful handling – the policy implies validators must enforce the context for those values[82]. For example, new transient status values (“running”, “failed”, etc.) were added as part of v2.1; these would be allowed as a minor update since they don’t break existing records, but they must be accompanied by validation rules tying them only to transient entity kinds.
- Adding New Fields or Record Kinds: The policy states that adding a completely new record_kind (beyond entity/edge/generator) is a MAJOR change and lists a rigorous process (design doc, schema update, updating all tools, migration plan, etc.)[86]. Even adding fields to existing records should go through proposal and review. The idea is to maintain schema integrity – no ad-hoc additions. It also sets expectations for how quickly changes can be rolled out (e.g. at least 2 weeks review/testing for a new record kind)[70][87].
Workflow for Schema Changes: A standardized workflow is described (Propose → Review → Implement → Test → Deploy) for any schema update[88][89]. It involves writing a design proposal, team review (ensuring backward compatibility and listing affected tools), then implementation (update schema, bump version, update validators & docs, prepare migration scripts)[90][89], followed by testing and deployment steps[91]. This formalizes how a change moves from idea to reality, enforcing that nothing goes live without proper versioning and stakeholder notification[92].
Critical Governance Declarations: The policy explicitly names the authoritative schema file path and calls it out as the single point of authority[74]. It also references other authoritative config files like DERIVATIONS.yaml for computed logic – meaning even derived fields must follow a governed spec. One notable rule: if documentation and schema conflict, documentation must be updated – this underscores that documentation is downstream of schema[8].
The policy’s classification as “AUTHORITATIVE” and status “Effective Date 2026-01-21” means it is in force and all current/future changes should abide by it[93]. This is crucial when auditing for compliance: any deviations (like an implementation not following the spec, or a new enum value used without bumping schema version) would violate this policy.
Violations & Enforcement: While the policy document itself doesn’t list specific past violations, it provides guidance to catch them. For example, if someone were to add a new entity_kind value “other” (which actually happened in v2.1) – that’s an open enum, allowed without a major bump, but must still go through the process (which it did, as we see it documented in Quick Reference v2.1). If a team added a field in the JSON output but not in the schema or vice versa, that would violate Principle 1.1 (Single Source of Truth)[8]. The existence of this policy likely arose from earlier inconsistencies (like multiple conflicting scope values, or documentation lagging behind code). It now provides a clear measure to identify such issues in our audit.
## SCOPE_AND_COUNTER_KEY_DECISION.md (SCOPE Decision Document)

### Purpose and context

 This is a critical decision record addressing the inconsistency of multiple SCOPE values and counter key formats across the project. It lays out the problem that the system is using two different scope codes (260118 vs 260119) in IDs and an all-new third code (720066) in configuration, as well as two formats for the counter keys (the underscore format vs the spec’s colon format)[94]. The document’s status “REQUIRES_DECISION” indicates it is meant to drive a leadership decision to choose a single authoritative SCOPE value and key format going forward. Essentially, it’s resolving a governance and configuration debt that could confuse the system if not unified.
Issue 1 – SCOPE Value Inconsistency: The document inventories the scope codes in use:
- 260118 – the original scope (presumably derived from the project start date 2026-01-18). It’s present in many legacy file IDs (15 files) and in the old registry snapshot[95]. For example, files like 0199900095260118_README.md have the 260118 suffix[96]. This scope appears in older documentation (e.g. examples in SEQ_ALLOCATOR_SPEC) as well[97].
- 260119 – a transition scope (one day increment from the original). This is used by a smaller set of files (5 files) and notably is the scope recorded in the current ID_REGISTRY.json (the active registry)[98][99]. For instance, 0199900006260119_CLAUDE.md has 260119 at the end[100][101]. The analysis suggests 260119 was an attempt to migrate (perhaps when the registry was rebuilt on Jan 19, 2026) but without clear rationale other than the date.
- 720066 – a new scope configured in IDENTITY_CONFIG.yaml (and used in recent documentation examples) but not yet used in any filenames[102][103]. The identity config’s scope: "720066" is marked as the “✅ CURRENT CONFIG” setting[102]. This value does not obviously correspond to a date; it may be an arbitrary or derived project code intended to replace date-based scopes. However, as of the decision doc, no file had been renamed to use 720066 yet. In other words, the config was updated, but the existing 20 files in the registry still carried the old scopes, creating a mismatch.
The document also notes “Other values” – e.g. 33 files with various odd scope values like 020001, 600002, etc., likely remnants of previous experiments or test data where the scope segment wasn’t consistent[104]. These seem to correspond to some IDs where the scope part might have been mis-assigned (possibly an earlier format using YYMMDD or other placeholders). They are outliers but contribute to the inconsistency.
Issue 2 – Counter Key Format Mismatch: The second half of the decision document highlights that the authoritative spec (SEQ_ALLOCATOR_SPEC) prescribes a counter key format of {SCOPE}:{NS}:{TYPE} (colon-separated, scope first)[36], whereas the current implementation and ID_REGISTRY.json use the format {NS}_{TYPE}_{SCOPE} (underscore-separated, NS first)[105]. This is evidenced by the registry content (e.g., keys like 999_01_260119) which conflicts with the intended 260119:999:01 format. The decision doc explicitly calls out this violation of the spec: one option is to change the code to match the spec, the alternative is to officially accept the underscore format and amend the spec[105][106].
Recommendations & Options: The decision document presents options to resolve both inconsistencies:
- For Scope: It strongly leans towards adopting 720066 as the single scope (Choice C)[107][108]. Pros: it’s a clean slate with no conflicts, and the config is already set to it; cons: all existing file IDs (20 files) would need to be renamed to the new scope, which is a one-time migration[108]. Alternatives were A) revert everything to 260118 or B) stick to 260119, but these perpetuate legacy or lack rationale[109][110]. The document notes 720066 is not meaningful yet, but a meaning can be assigned; by contrast 260118 has meaning (project start date) but is legacy, and 260119 is just an odd artifact. The recommendation (✅) is 720066, marking it clearly as the preferred path[108].
- For Counter Key Format: The recommendation is to switch to SCOPE:NS:TYPE (colon-separated) to align with the spec[111]. Rationale: it matches the ID segment order and what the contract states, and colons are a standard for hierarchical keys[36]. The alternative of keeping NS_TYPE_SCOPE (underscores) is noted as violating the authoritative spec and grouping by NS (which is less logical since NS is project-specific)[112]. The decision doc clearly prefers the spec-compliant format, noting a code change in registry_store.py would be needed (line 194)[105].
Implications and Migration: The document provides a Decision Matrix of options A, B, C for scope and formats, with Option A (Scope=720066 & Format=SCOPE:NS:TYPE) marked as Recommended[113][114]. It also outlines an implementation checklist for the recommended option: update code (change the key format in the store code, search for any usage of the old pattern)[115], update the registry file (back it up, then transform all counter keys and change the scope field from 260119 to 720066)[116], and update documentation (e.g. the SEQ_ALLOCATOR_SPEC examples, README, and Quick Start Guide) to use 720066 and the new format[117]. This shows a thorough approach to enforce consistency across the board once the decision is made.
Status & Authority: As of the document’s date (2026-01-21), it requires a decision – meaning the changes might not yet be applied, but this document itself becomes an authoritative reference once approved. It is labeled a “CRITICAL_SYSTEM_DECISION”[118], highlighting the importance of resolving this now to avoid further divergence. In terms of governance, this decision is a direct application of the Schema Authority Policy: it’s addressing a case of split-brain configuration (multiple scopes) by re-establishing a single source of truth (one scope)[7], and resolving a spec vs implementation conflict by either aligning implementation to spec or vice versa (with the recommendation to align to spec, maintaining the authority of the spec)[36][112].
In summary, the SCOPE decision document is essentially an audit of inconsistencies itself, and a directive to fix them. It prioritizes 720066 as the unified scope and {SCOPE}:{NS}:{TYPE} as the unified counter key format, which our cross-document audit will reflect.
## 16_DIGIT_ID_BREAKDOWN.md

### Purpose and context

 This document provides a human-friendly breakdown of the 16-digit ID format, explaining each component (Type, Namespace, Sequence, Scope) in simple terms with examples. It appears aimed at onboarding or decision-makers to quickly understand the ID structure and the current scope dilemma in layman’s terms (it even uses emojis like 1️⃣ 2️⃣ 3️⃣ 4️⃣ to denote sections). Its context is likely to support discussions (like the SCOPE decision) by clarifying what each part of the ID means and why consistency matters.
Key Concepts Explained: The document divides the 16-digit ID into 4 labeled sections and explains each:
TYPE (Digits 1–2) – File type code. It explains that these two digits indicate the kind of file (e.g., 01 = Markdown, 20 = Python, 11 = JSON, etc.) and gives a table of examples[119]. For instance, 01 corresponds to a Markdown file (.md), 20 to a Python file (.py), 12 to YAML, and so on, including 00 as Unknown/Other for any file that doesn’t match a known extension[119]. The “Why it exists” is to let you identify file type from the ID at a glance[120]. This section essentially mirrors the type_classification from the identity config, but in documentation form.
NS (Namespace, Digits 3–5) – Folder or category code. It describes this 3-digit code as representing which directory or logical group the file belongs to[121]. An example: 999 is shown as “Uncategorized (fallback)” for files that don’t fit any specific folder rule[122]. A table lists codes like 100 for Documentation (docs/**/*), 110 for Data files (data/**/*), 200 for Python scripts, 201 for PowerShell scripts, 300 for Storage, 400 for Assets, 410 for Logs, 420 for Reports, and 999 for Uncategorized, each with an example ID illustrating the code in context[123][124]. This directly reflects the namespace_routing rules from identity config (and matches what we saw in counters keys). Why it exists: to group files by purpose/location, making it easy to tell from the ID what part of the project the file is in[125].
SEQ (Sequence, Digits 6–10) – The 5-digit sequence number for uniqueness[126]. The doc explains this as the counter that increments for each new file in a given type/namespace category[127]. Examples: 00001 for the first file, 00095 for the 95th file, etc., inserted into an ID (e.g., 0199900**00095**260118_README.md has “00095” indicating it was the 95th file of that type/namespace)[128][129]. It also shows 00495 as an example for 495th file, and 99999 as the max possible[128]. The “Why it exists” is clearly stated: to ensure every file gets a unique number, prevent collisions, and to only ever increase (monotonic)[127]. It notes that each (TYPE, NS, SCOPE) combination has its own independent counter[130][131]. For example, Markdown in docs has its own sequence range separate from Python in scripts[131]. This aligns perfectly with the SEQ_ALLOCATOR_SPEC rules (monotonic counters per key)[132].
SCOPE (Digits 11–16) – The project/system identifier code[133]. It explains that this 6-digit number denotes which project the file belongs to, acting like an “area code” or “project stamp” so IDs from different projects won’t clash[134][135]. The example given: 260118 corresponds to “This project, started January 18, 2026”[136]. ## FILE_SCAN_CSV_DERIVATION_SPEC — Complete Column Derivation Documentation

### Purpose and context

 This document is a detailed specification of how each output column in the File Scan CSV registry is derived. In the Directory Management System, the “file scanner” produces a CSV or JSON of all files and their identity attributes; this spec ensures that any implementation of a scanner does so in a consistent, deterministic way. It’s labeled as a target contract (DOC-REG-FILE-SCAN-CSV-DERIV-001) version 1.0.0[165], meaning it defines the rules that must be followed to comply with the registry format REG-SCAN-CSV-1.0. Essentially, if different scanners or tools generate the registry, they all must derive values the same way as per this document.
Key Concepts & Structure: The spec is organized into sections corresponding to the derivation process:
Inputs: It enumerates all inputs considered authoritative for derivation[166]. This includes runtime inputs (the scan_root directory being scanned, a scan_timestamp_utc, and a scan_id unique run identifier)[167]. Also, configuration inputs such as an optional IDENTITY_CONFIG.yaml and scan behavior settings[168][169]. The identity config, if present, can override default identity rules – fields like id_regex (regex to detect existing IDs, default ^\d{16}_), placeholder_id (default 000...0000 16 zeros), default_scope (default "GLOBAL" if not set, though in this project they set a numeric scope), and mappings for type codes, namespace codes, and scope by namespace[170]. All those are listed with defaults, indicating the scanner will use those rules to parse filenames and assign codes[170]. Additionally, general scan config like which directories to exclude, filename patterns to include/exclude, whether to treat directories as records, etc., are inputs (the spec notes this contract assumes directories are included as rows)[169].
Execution Model: It describes the algorithm of the scanning process in precise steps[171][172]. For example, in Run initialization[173]: normalize scan_root to an absolute path, set first_seen_utc = now_utc() once per run[174], generate a scan_id (unique, with recommended format either timestamp + random hex or a UUID4)[175], then determine effective identity settings by either using the identity config values or falling back to the defaults (the list from ID_REGEX to SCOPE_MAP)[176]. This step essentially merges config overrides with defaults to fix the settings for the run.
Then Traversal rules[177][178]: recursively walk the scan_root, skipping anything in an excluded directory, skipping output directories if within root, etc.[178]. For each file, apply include/exclude patterns (only include if it matches includes, and skip if matches any exclude)[179]. For every included file or directory, emit exactly one row in the output (even if errors occur, in which case error columns are filled)[180]. These rules guarantee that the scan is deterministic and consistent (everyone will skip or include the same files given the same config).
Column Derivation Catalog: The heart of the document is a catalog of every column and how to compute it[181]. It’s divided into subsections by column grouping:
Shared Helper Rules: It defines helper functions H1, H2, ... H10 which are used by multiple column derivations[182][183]. For example, now_utc_iso8601_z() to get current time in Z format[184], to_utc_iso8601_z(epoch) to format epoch timestamps[185], sanitize_code_token(s) to produce a valid code (uppercase and alphanumeric with underscores)[186], relative_path_from(scan_root, path) to get a normalized relative path with forward slashes[187], extension_from_name(name) to extract a file extension in lowercase[188], match_id_prefix(name) to check if a filename starts with an ID per ID_REGEX[189], extract_16_digit_prefix(name) to actually capture the 16-digit ID if present (always using the fixed ^\d{16}_ pattern regardless of a broader ID_REGEX, to keep doc_id stable)[190], guess_mime_type(name) using the Python mimetypes library[191], permissions_from_stat(st_mode) to get Unix-style file permission bits[192], and sha256_file(path) to compute file hash (with instructions on chunked reading)[193]. Each helper outlines exactly how to handle edge cases (e.g., return '' if no mime type, or raise errors for permission issues which then populate error fields)[194][195]. These helpers standardize common computations so that, for instance, every scanner will compute SHA-256 in the same way and format.
Per-Run Columns: Columns that have the same value for all rows in a scan. For example, scan_id – the rule is “same value for all rows, never blank”[196]. Similarly scan_root path, scan_timestamp (first_seen_utc) would be set once and applied to each record. The spec would list each such column with Source and Rule. We see scan_id as one example[197], and presumably scan_root, scan_timestamp_utc (first_seen) follow.
Per-File Columns: This includes identity-related columns and file metadata. While not shown in our snippet, likely columns like id (the 16-digit ID if any), has_id_prefix (bool if file already had an ID in name), needs_id (bool if it needs an ID allocated), type_code, ns_code, scope (which would be the default or overridden scope name, e.g. “GLOBAL” or project code), filename, extension, size_bytes, mtime_utc, mime_type, sha256, etc. Each column entry in the catalog specifies exactly how to derive it. For example, doc_id might be “if match_id_prefix(name) true, then extract_16_digit_prefix(name), else blank or placeholder” – thereby pulling the ID if present. type_code derivation: look up the file’s extension in the TYPE_MAP (from identity config) – if found use that code, else use fallback code (like "00")[198][199]. ns_code: apply NS_MAP rules by checking the file’s relative path against each pattern in order – the first matching rule gives the ns_code, otherwise default to 999[24]. scope: probably simply the configured default_scope from identity config (or "GLOBAL") – in EAFIX’s case, the identity_config provides a specific numeric scope, so that will be used. has_id_prefix: use helper H6 match_id_prefix[189] to set a boolean. permissions: call H9 to get perm bits if it’s a file, etc. errors: if any step fails (e.g., file can’t be read for hash), the spec likely has columns for error_type or similar, which are set if an exception is caught (the spec mentions “populate error fields” if errors occur)[180].
Because the document is titled “Complete Column Derivation Specification,” it presumably covers all columns that appear in the CSV output of the scan. This correlates with the Registry v2.1 columns, though as of v1.0 of this spec, it might be focusing on file entity records only (26 columns were mentioned in a quick start guide)[200]. Indeed, a quick-start snippet noted the scanner outputs 26 fields including type_code, ns_code, scope, has_id_prefix, needs_id, etc.[200]. Those derivations are exactly what this spec would define.
Critical Values & Defaults: The spec explicitly lists default values for identity config fields[170], which are important: id_regex = '^\d{16}_' (so by default, any 16-digit prefix followed by underscore is considered an ID), placeholder_id = '0000000000000000' (16 zeros as a dummy ID for empty slots), default_scope = 'GLOBAL' (used if no scope is configured – in EAFIX they override this with a numeric scope), and empty maps for type_code_by_extension, etc., meaning if not specified, no special mappings (everything would fall to "00" or "999"). These defaults ensure the scanner can run even without a config file, albeit in a generic mode.
Another critical aspect is the generator_intent: the front matter says “Any scanner that claims to produce REG-SCAN-CSV-1.0 MUST implement these derivations exactly.”[201]. This makes the document an authoritative contract for generation logic. It complements the JSON Schema (which covers validation rules) by covering transformation rules. The note explicitly states that validation rules belong in the schema, whereas this doc is about derivation logic[202] – enforcing separation of concerns (and aligning with the Schema Authority Policy where DERIVATIONS.yaml is authoritative for computed fields[203] – this spec is effectively the human-readable form of such derivation rules).
### Versioning and authority

 As mentioned, it’s version 1.0.0 and status “target_contract”[165]. “target_contract” likely means it’s the agreed standard that implementations should meet. Its applicability is to REG-SCAN-CSV-1.0 (meaning the first version of the scanning output format)[201]. If the registry format evolves (v2.1, etc.), presumably a new version of this spec might be issued if column derivations change or new columns are added. The spec serves as an authoritative source for how to compute all non-trivial fields. For example, everyone should compute sha256 the same way, and use the same method to extract doc_id from filename. Any deviation (like using a different regex, or not uppercasing rel_type if that were in scope here) would cause inconsistencies in the registry.
In summary, FILE_SCAN_CSV_DERIVATION_SPEC ensures that the data recorded in the registry is consistent regardless of who or what generated it. It encodes the logic that, combined with the identity config and schema, guarantees the single source of truth nature of the registry. For our audit, this spec is useful to identify if any derivation logic in practice isn’t following spec (e.g., the counter key format difference is one such logic mismatch, though that’s more on allocation side; another could be if a scanner didn’t correctly detect an existing ID and created duplicates – which this spec attempts to prevent by the id_regex method). Thus far, no glaring conflicts with this spec have been noted, which is good. It’s meticulously documented to avoid such issues.

## Inconsistencies and Conflicts Across Documents
After reviewing the documents, several cross-document inconsistencies and legacy issues emerge. Below we identify each major conflict or discrepancy, explain it, and cite the evidence:
Multiple SCOPE Values (260118 vs 260119 vs 720066): There is a clear conflict regarding the 6-digit project scope code. The old registry and many file IDs use 260118 (derived from the Jan 18, 2026 start date)[95], while the current active registry (ID_REGISTRY.json) was initialized with 260119 (Jan 19, 2026) as scope[204]. Meanwhile, the latest configuration file sets 720066 as the intended scope[102], a value not yet reflected in any filenames. This leads to a split-brain situation where the identity configuration does not match the actual IDs in use. The Scope Decision document explicitly calls this out as a critical issue, noting 15 files on 260118, 5 files on 260119, and config on 720066[95][205]. The Schema Authority Policy demands a single source of truth – having two active scope codes violates that principle[7]. It also undermines the “never change once set” guideline for scope[135]. The decision recommendation is to converge on one scope (720066) and rename all files accordingly[108], which implies the current state is indeed inconsistent and must be remediated. Until fixed, tools like the scanner might treat “GLOBAL” vs numeric scope differently (the identity config default_scope “GLOBAL” is overridden by 720066 now, but existing IDs contradict that). This inconsistency is a governance policy violation (scope drift) and a practical problem (it required marking many IDs with legacy_scope in the registry to reconcile the mismatch)[6].
Counter Key Format Mismatch (Spec vs Implementation): The authoritative SEQ_ALLOCATOR_SPEC.md stipulates that counter keys should be stored as "SCOPE:NS:TYPE" strings[206], and even gives an example "260118:200:20"[33]. However, both the older registry snapshot and the current registry use the format "NS_TYPE_SCOPE" with underscores (e.g., 999_01_260119)[3][207]. This is confirmed in the decision doc, which notes the spec’s format and labels the underscore format as a divergence that “violates the authoritative spec”[208][112]. The inconsistency likely arose because the initial implementation followed a different convention than the written spec. This is a direct spec compliance issue – the system is not following the agreed contract. Under the Schema Authority Policy, the spec (or rather the design contract) should be updated or the code fixed; the decision document recommends fixing the code to use SCOPE:NS:TYPE as the key[111]. Until that happens, there’s effectively a documentation vs implementation conflict. It’s worth noting that this format difference, while internal to the JSON structure, can lead to confusion or errors if someone assumed a different grouping. For example, grouping counters by scope first (as spec intends) vs by namespace first (as currently done) could impact how data is migrated or displayed. This conflict is logged as an issue requiring immediate resolution[94] and is an actionable governance violation (the authoritative contract from SEQ_ALLOCATOR_SPEC is being broken in practice).
Identity Config vs Documentation Discrepancies: The IDENTITY_CONFIG.yaml in use has scope: "720066" configured (current truth)[102], but much of the documentation and example usage is out of sync with that. For instance, the Quick Start Guide excerpt still showed an example config with scope: "260118"[209], which was the old value. Also, the 16-digit breakdown and scope decision explanation documents highlight that the config’s scope (720066) isn’t reflected in actual file IDs yet[205][150]. This means developers reading the docs might see conflicting info: some docs (system documentation, etc.) present 720066 as the project scope (as a key metric)[210][211], while the files and older guides show 260118/260119. This inconsistency can cause confusion about which scope to use when allocating new IDs. It violates the expectation from Schema Authority Policy that documentation and config should be aligned with the source of truth (in this case, identity_config plus registry)[8]. The decision doc’s checklist includes updating all documentation (guides, README, spec examples) to use 720066 once the decision is finalized[117], indicating acknowledgment of this drift. Until those docs are updated, there’s a risk of someone following an outdated guide and inadvertently using the wrong scope or assuming an incorrect configuration.
Legacy ID Prefix in File Names vs New Schema Fields: In the current system, the 16-digit ID is embedded in the filename itself. The Unified Registry proposal introduces explicit doc_id, filename, and extension fields to separate the ID from the name[212][213]. In practice, Phase 1 did not rename files or remove prefixes – it simply documented how to extract them. Thus, we have a transitional situation where file names carry legacy IDs (which is fine) but the new schema expects those IDs to also be recorded in separate fields. The registry v2.1 Quick Reference confirms that doc_id is now a distinct column (to hold the 16-digit ID) and filename is stored without the prefix[212]. However, in the actual JSON registry file (which is still essentially v1.x format), we don’t yet see a separate doc_id field – the ID is only in the id field (which doubles as record_id for file entities) and in the file_path string. The older registry snapshot did not store doc_id separately either (it listed id and file_path). This means until the registry format is migrated to 2.x, the system relies on parsing the file_path to get the ID. This isn’t so much an error as a legacy convention still in use: the practice of prefixing filenames with the ID is a carryover to maintain until new tools can handle ID-as-field. It could be seen as a temporary violation of the new schema’s spirit (since ideally filename should not include the ID, according to new design). The File Scan derivation spec even notes that extraction of the 16-digit prefix is always done by a fixed regex[214], implying the system is aware of this legacy format and deterministically handling it. As the unified approach rolls out, we expect file renaming might stop (or be optional) and IDs purely managed in registry. But for now, this discrepancy is something to manage carefully to avoid double-counting or confusion (e.g., the same ID showing up as both id and part of file_path). It’s a controlled inconsistency during migration.
Deprecated/Unused Enum Values and Fields: With the expansion to v2.1, some enum values and fields are defined that currently have no usage in the active data (they are forward-looking, or placeholders from previous ideas):
The entity_kind enum now includes "other" (for unforeseen entity types)[151], but as of now, no records likely use “other” – it’s there for flexibility and governance (with policy imposing limits on using “other”)[215]. Similarly, entity_kind includes values like “module” or “process” in v2.1, yet the system hasn’t implemented module_id or process tracking in Phase 1 (those fields are always blank or not present). They are effectively defined but unused until a future phase uses them. This isn’t harmful, but it’s a point of potential confusion – e.g., documentation shows entity_kind: "process" as allowed, but no actual process records exist yet.
The status enum added values for transients (“closed”, “running”, etc.)[216] which currently would not apply to any record (since entity_kind: transient records aren’t in use yet). Those are essentially reserved for Phase 3. We should ensure tools don’t mistakenly apply them to normal files. The Schema Authority Policy flags status as conditional – meaning such values must be validated in context (only allowed if record is transient)[82]. If the current validators do not yet enforce that (because transients not implemented), there’s a latent risk of misuse. However, since no transient records exist, it’s just an unused extension for now.
Fields like module_id, process_id, role_code were mentioned in the unified analysis and appear in the column lists[62], but in the current registry they do not exist (v1.x) or are always empty (if v2.1, they’d be new but unpopulated for files). In the older registry JSON, we saw module_id and process_step_id listed in the columns definition but not actually present in any record[11][217]. This suggests those fields were anticipated but not used – effectively dead fields at that time. They might have been prototypes for Phase 1 that got deferred. Their presence in the schema without usage could be considered clutter or a minor governance lapse (introducing fields too early). According to policy, adding fields is allowed in a minor version, but ideally they’d be used. If not, they should at least do no harm; we saw none of the records in 1.1 snapshot had non-null values for them. It’s an example of how some legacy or experimental schema elements persisted.
The namespace codes table shows many values (100,110,200,...,999) in use[123]. However, the decision doc noted 33 files had “other” scope values like 020001, 600002 etc. It’s possible those weird scopes were actually mis-encodings of what should have been namespace or type codes. For example, 020001 might have been an attempt to encode something in the scope portion (perhaps “02” type and “001” ns mistakenly put into scope field). These might indicate a legacy format bug or deprecated ID pattern. If those 33 files are still around, they represent IDs not conforming exactly to the TTNNNSSSSSSSSSSS breakdown (maybe an older version with YYYYMM in scope). The plan would likely be to reallocate or deprecate those. This is an inconsistency within the dataset itself. While not directly a doc-to-doc conflict, it’s a cross-check issue flagged by the decision document[104] as something to resolve (they categorized them as “Other values” that don’t fit the standard). This highlights that some legacy IDs might not obey the current schema, violating the ID format validation rule (must match ^\d{16}$ with correct segmented meaning)[218]. They need cleanup or grandfathering.
### Documentation vs implementation timing

 There is a slight discrepancy in timing where documentation (like the Registry v2.1 Quick Reference and Unified Analysis) assumes certain features (edges, assets, etc.) that the implementation hasn’t caught up with. This is not a fault per se, since those docs are forward-looking. But in an audit it appears as a gap: e.g., Quick Reference lists absolute_path and directory_path as new entity columns[162], yet the current scanner output (REG-SCAN-CSV-1.0) likely doesn’t include those (the quick-start mentioned 26 columns, which did not include directory_path or absolute_path). So there’s a version skew: the documentation of v2.1 is slightly ahead of the actual deployed scanner version (which seems to be v2.0/Phase1 in terms of output). For instance, the File Scan spec (targeting REG-SCAN-CSV-1.0) doesn’t mention directory_path or module_id since those come in with v2.1. If someone were to compare the spec and the quick reference blindly, they might notice the spec doesn’t generate all 80 columns (only ~26). This is by design (different versions), but it underscores the need for version alignment. It’s important that once the Phase 2+ features are implemented, the File Scan Derivation Spec is updated to cover the new columns and perhaps revved to REG-SCAN-CSV-2.0, to avoid confusion. Right now, this is a known transitional inconsistency: documents describing v2.1 vs actual system at v2.0. The governance policies do allow phased rollout, but they emphasize backward compatibility and documentation for transitions[67][71]. We see that happening (Phase 1 is backwards-compatible; Phase 2 to come). As an audit point, stakeholders should be aware which docs apply to current state vs future, to prevent misimplementation.
### Governance policy violations

 Summarizing the above in terms of the Schema Authority Policy: the biggest violation was the spec vs implementation divergence (counter key format)[208] and the documentation drift (multiple scopes, outdated examples)[219][209]. The Single Source of Truth principle was bent by having identity_config.yaml say one thing and the registry using another scope, which is precisely the kind of situation the policy tries to avoid. This has been recognized and is being rectified via the decision doc. Another governance aspect is the introduction of new enum values (status and entity_kind) – according to policy, adding open enum values is allowed without a major bump, but conditional enums like adding transient statuses likely required careful consideration (the policy says adding values to conditional enums still requires validators)[220]. We should verify that the JSON Schema was indeed updated to include those new values and that any validator logic for status was updated. If not, that’s a latent policy violation (e.g., if schema allows them but validator isn’t checking context). There’s no explicit evidence in docs of a missed update, so presumably it was handled. Finally, the presence of legacy IDs marked with legacy_id: true and legacy_reason: scope_mismatch in the registry[6] indicates the system has a procedure to flag governance exceptions (those entries violate the current scope rule, hence marked). This is good practice – it doesn’t fix the root issue but at least documents it. However, it’s essentially an admission of a governance policy violation in the data (multiple scopes), which further underscores the importance of the SCOPE decision to eliminate these legacy exceptions.
### Outdated conventions and legacy data

 The audit uncovered a few practices that are essentially holdovers from earlier stages:
Using date-based scope values (260118, 260119) is an outdated convention as the project moves to a non-date arbitrary code (720066). The documentation provided analogies (area code, etc.) to justify moving away from date-based if needed[221][150]. We consider 260118/260119 as legacy scope format now. The recommended practice will be a fixed project code not tied to date (more flexible if project spans multiple days).
The underscore-separated counter key (NS_TYPE_SCOPE) is a legacy of initial implementation; the new convention is colon-separated (SCOPE:NS:TYPE) which is more logical and standard[208]. Until changed, the code and data still reflect the old format.
The fact that 33 files have irregular scope values like 020001 suggests an even older ID formatting rule (perhaps from an early iteration of the system) that has since been deprecated. Those IDs might contain encoded dates or incorrect segment placements. This is legacy data that does not conform to the final 16-digit breakdown as intended, and should be corrected or at least isolated. The system likely treats them as quirks – they could even be test files not in production use. Regardless, they represent an outdated ID format no longer generated.
Another possible outdated element: the identity config default default_scope: "GLOBAL" hints that originally the system might have considered using a literal “GLOBAL” as a scope for single-project scenarios. In practice, the project chose numeric scopes. “GLOBAL” is not used anywhere in the actual registry; it’s essentially a deprecated default. Now that a specific scope is always configured, “GLOBAL” as a concept is moot. It remains in the spec as a fallback but in EAFIX context it’s unused. One could consider that a retired idea (the docs don’t mention “GLOBAL” scope anywhere except the derivation spec default).
Unused fields like process_step_id in the 1.1 registry snapshot[11] appear to be an old concept (perhaps they thought of tracking process steps separately from process id, then dropped it). This field does not show up in v2.1 plans (they have process_id and module_id but not process_step). So process_step_id in that snapshot looks like a relic of a design iteration that didn’t persist. It’s essentially a ghost field now and can be ignored moving forward.
In conclusion, the cross-document audit reveals a system that is well-documented and largely consistent in theory, but has some transitional inconsistencies due to evolving from legacy conventions to newer standards. Key conflicts like the scope and counter key issues have been identified by the team and are in the process of resolution[94][111]. Ensuring the identity config, registry implementation, and all documentation align on one scope (720066) and the spec-prescribed formats will resolve the biggest inconsistencies. After that migration, the remaining discrepancies (unused enums, placeholder fields) are mostly harmless and expected as part of extending the schema for future use – though they should be monitored under the Schema Authority Policy to remove or solidify them in future versions. By adhering to the Schema Authority Policy going forward – i.e., letting the JSON Schema and spec drive changes and promptly updating all artifacts – the project will maintain the Single Source of Truth without the kinds of conflicts noted in this audit[7][8].
