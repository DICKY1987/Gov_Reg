# Capability Mapping System

## Overview

The Capability Mapping System is a 4-step automated pipeline that classifies every Python file in the Gov_Reg repository and tags capabilities into the SSOT (Single Source of Truth) registry.

**Status:** ✅ Complete as of 2026-02-20
- **574 files tagged** with capabilities
- **796 patch operations** applied to SSOT registry
- All pipeline stages validated and producing consistent outputs

---

## System Purpose

The pipeline performs intelligent Python file analysis to:

1. **Discover** capabilities across the codebase using AST and call-graph analysis
2. **Inventory** all Python files with metadata (LOC, imports, dependencies)
3. **Map** files to purposes based on extracted features and classification rules
4. **Promote** the mappings to the SSOT registry as structured patches

This ensures every script's role is documented, searchable, and maintained as the codebase evolves.

## Pipeline Steps

### Step 1: Capability Discovery
**Module:** `capability_discoverer.py`  
**Input:** Repository root  
**Output:** `CAPABILITIES.json` (all discovered capabilities)

Scans all Python files and extracts:
- Argparse command-line capabilities
- Call graph relationships
- AST-based function/class signatures

### Step 2: File Inventory
**Module:** `file_inventory_builder.py`  
**Input:** Repository root + CAPABILITIES.json  
**Output:** `FILE_INVENTORY.json` (metadata for each Python file)

Builds a complete inventory with:
- Lines of code
- Import statements
- Module dependencies
- File path and metadata

### Step 3: Purpose Mapping
**Module:** `purpose_registry_builder.py`  
**Input:** FILE_INVENTORY.json + COMPONENT_CAPABILITY_VOCAB.json  
**Output:** `FILE_PURPOSE_REGISTRY.json` (files → purposes)

Applies classification rules using:
- Keyword matching in filenames/paths
- Import pattern analysis
- Manual overrides from configuration
- Multi-purpose classification support

### Step 4: Registry Promotion
**Module:** `registry_promoter.py`  
**Input:** FILE_PURPOSE_REGISTRY.json + SSOT registry  
**Output:** Patches applied to `01999000042260124503_REGISTRY_file.json`

Generates and applies structured JSON patches to SSOT:
- Adds `capabilities` array to file records
- Preserves all existing registry data
- Creates evidence trail and audit logs

## Key Files and Locations

### Configuration Files
- **`COMPONENT_CAPABILITY_VOCAB.json`** (in schemas/)  
  Defines the capability taxonomy and classification rules
  
- **`SCRIPT_CLASSIFIER_OVERRIDES.json`** (in mapp_py/)  
  Manual overrides for edge cases and exceptions

### Output Locations
All pipeline outputs are written to `.state/purpose_mapping/` (unless overridden):
- `.state/purpose_mapping/CAPABILITIES.json`
- `.state/purpose_mapping/FILE_INVENTORY.json`
- `.state/purpose_mapping/FILE_PURPOSE_REGISTRY.json`

### Evidence and Audit
- `.state/evidence/purpose_mapping/` — step-by-step evidence trail
- `.state/evidence/purpose_mapping/registry_patch_*.json` — applied patches

## Maintenance

### Adding New Capabilities
1. Edit `COMPONENT_CAPABILITY_VOCAB.json` to add new purpose definitions
2. Update `purpose_registry_builder.py` classification rules if needed
3. Re-run step 3 and 4 to update the registry

### Debugging Classification Issues
1. Check `.state/evidence/purpose_mapping/` for step-by-step logs
2. Add manual overrides to `SCRIPT_CLASSIFIER_OVERRIDES.json`
3. Re-run step 3 to apply overrides

### Updating the SSOT Registry
The system is designed to be **idempotent**:
- Re-running step 4 will generate fresh patches
- Existing capability tags are preserved (no duplicates)
- Use `--force` flag to override existing tags
