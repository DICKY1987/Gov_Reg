Solutions for Remaining Development Steps to Achieve a 100 % Functioning Registry Automation System
Introduction

The registry automation system has evolved from a script‑centric bug list into a contract‑gap‑driven, template‑native plan. However the runtime still functions as a toolkit rather than a fully governed pipeline. Achieving a 100 % functioning version requires closing the remaining gaps in pipeline logic, data contracts, identity promotion, orchestration infrastructure and operational automation. This report lays out concrete solutions for each remaining development step.

1. Fix the core pipeline to reliably produce patches

The existing pipeline_runner can silently skip valid transform output, resulting in no patches. To close this gap:

Remove non‑existent checks. Delete the conditional that waits for result["data"]; instead treat any non‑None transform result as the patch candidate. This ensures that patched records are always staged.

Validate the result envelope. Ensure the transform returns a predictable structure (e.g., {"patches": […], "evidence": …}) and add explicit error handling if the structure is wrong. Use rule‑based validation patterns (data type, range and format checks).

Add regression tests. Create unit tests that run the pipeline with known transforms and verify that patches are produced and contain expected changes. Add negative tests to confirm the pipeline fails when transform results are malformed.

2. Close the identity promotion gap (sha256 → file_id)

Identities are promoted only when both a valid file_id and sha256 exist; many records remain stuck in a “py_*" state. Solutions:

Backfill sha256 values. Write a backfill script to compute missing sha256 hashes for each file. Use a strong cryptographic hash (e.g., SHA‑256) so that files with identical content produce the same fingerprint and deduplication is reliable.

Build a sha256 → file_id mapping. Maintain an authoritative mapping file (SHA256_TO_FILE_ID.json) that assigns a 20‑digit file_id to each unique sha256. During promotion, look up the record’s sha256 and insert the corresponding file_id. Fail the promotion if no mapping exists.

Make unresolved identities fail‑closed. Modify the mutation runner so that any record lacking a valid file_id stops the pipeline. Test this with fixtures covering cases of missing sha256, missing mapping entry and collisions.

3. Implement defaults and null‑handling

Defaults and null‑coalescing currently do nothing because the column dictionary exposes default_value = None. In production, null handling is a key validation challenge. The solution:

Create an authoritative defaults catalog. Extend the column dictionary to specify meaningful defaults (e.g., [] for list fields, 0 for integer counters, explicit sentinel values for unknown strings). Each default should align with business semantics and be defined in configuration rather than code.

Update column_loader. Load these defaults when reading dictionary definitions. Expose them as default_value so other modules can access them.

Enable the default injector. Modify default_injector to populate missing fields with the default from the catalog. Log whenever a default is injected to aid auditing.

Strengthen null_coalescer. Replace null or empty strings with meaningful defaults as per configuration. Provide options to interpret the string "null" or empty strings as actual null values and to specify replacements.

Validation and monitoring. Add validation rules to ensure required fields are never null and optional fields follow documented defaults. Report null values explicitly so operators can distinguish between valid missing data and pipeline defects.

4. Strengthen end‑to‑end validation

Current validation checks only schema presence and existing validators. A robust ETL should enforce data quality along multiple dimensions. Implement:

Rule‑based field validation. Define rules for data types, ranges, formats and business logic. Validate during transformation to catch issues early.

Automated validation at each stage. Schedule validation jobs that compare source and target record counts, run sampling and statistical checks and alert on failure.

Error handling and classification. Use explicit error logging, categorize errors by severity and implement thresholds for failing or tolerating minor issues. Provide rollback mechanisms if validation fails.

Continuous validation framework. Adopt a framework such as Great Expectations or dbt tests for declarative expectations; integrate them into the pipeline so that rules are version‑controlled and produce machine‑readable test reports.

Data quality dimensions. Verify correctness, completeness, validity, consistency and uniqueness. For example, check referential integrity (edges linking valid identifiers) and uniqueness constraints (no duplicate file_id).

5. Fix timestamp authority in clustering

The timestamp clusterer uses the wrong field (created_at) instead of authoritative registry fields (e.g., created_utc, updated_utc). Solutions:

Parameterize the clusterer. Read the authoritative timestamp field from configuration. Allow selecting between created_utc or updated_utc depending on context.

Repair existing logic. Remove assumptions about created_at; update queries to use the configured field. If multiple timestamps exist, implement logic to choose the earliest or most authoritative.

Test across edge cases. Use synthetic datasets with mixed timestamp patterns to verify that clustering groups records correctly after the fix.

6. Turn doc ID resolution and module dedup into governed mutations

Currently the resolvers detect collisions and report them but do not apply fixes. Deduplication must include detection, matching and merging. Action items:

Create a dedup match engine. Use detection heuristics (matching normalized module names, paths and file hashes) to group duplicates; then choose an authoritative record (e.g., the earliest or most complete).

Produce a mutation plan. Generate patch operations that merge duplicates: update doc_id of duplicates to the authoritative one, re‑point edges and entries, and remove orphaned nodes.

Integrate into the mutation runner. Execute the dedup mutation plan under the same governance as other write operations. Add idempotent checks so repeated runs do not duplicate work.

Validate with tests. Build fixtures with duplicate modules and doc IDs and confirm that after running the resolver the registry contains exactly one canonical entry.

7. Clean registry truth and remove path pollution

FILE WATCHER prefixes and other pollution remain in registry truth. Detection occurs at analysis time but the source data is not fixed. Solutions:
