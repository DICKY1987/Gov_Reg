# Detailed Solution Document for CLI Application

## Gov_Reg Audit Hardening, Authority Resolution, and Gap-Closure Execution Plan

## 1. Purpose

This document defines how a CLI application should modify the current audit and verification process so it stops confusing discovery with closure, identifies real canonical authorities, repairs blocker conditions, and produces a defensible gap-closure assessment for the Gov_Reg ID / registry system.

The immediate problem is not only missing files. The bigger problem is that the current audit treats â€śstrongest file foundâ€ť as if that proves â€ścorrect canonical authority,â€ť even when the same audit also reports broken imports, duplicate implementations, conflicting orchestrators, registry-shape drift, and unresolved support-file mismatches.  

The CLI application must solve that by doing four things:

1. harden the audit model
2. verify execution and authority instead of only file discovery
3. repair or isolate known blocker conditions
4. produce a final authority map and remaining-gap report that is actually trustworthy

---

## 2. Core Problem Statement

The current audit has several failure modes that must be corrected.

### 2.1 Discovery is being mistaken for closure

The review correctly found that the audit marks some components as `found` even while also documenting dependency breakage, conflicting implementations, or unresolved canonicality. That is a reporting defect. 

### 2.2 The final assessment contains a logic contradiction

The audit says `has_single_canonical_runtime: true` while also saying `has_single_canonical_orchestrator: false`. That is not a coherent state description. At best, the repo has a dominant runtime nucleus with contested canonical status.  

### 2.3 Canonicality is not actually proven

The audit often recommends authority based on the best candidate found, but it does not consistently prove:

* imports resolve
* runtime can initialize
* the dominant orchestrator actually uses the candidate
* the file is contract-aligned
* the file is not shadowed by another current implementation 

### 2.4 The repo still contains blocker-level defects

The audit identified several concrete blockers:

* missing `ID_SCRIPT_INVENTORY.jsonl`
* `WRITE_POLICY_IDPKG.yaml` mismatch
* broken reconciler import in `healthcheck.py`
* unresolved `dir_id_handler` version mismatch
* duplicate watchers and duplicate pipeline runners
* missing dedicated lookup/join resolver
* missing dedicated unified column runtime engine 

---

## 3. Required Outcome

The CLI application must transform the current state into one where the repo can answer, for each system responsibility:

* what file is the canonical authority
* whether that authority is executable
* whether it is actually used by the dominant runtime
* whether it is aligned with the active identity contract
* whether any competing implementations remain
* whether the corresponding gap is truly closed, still partial, or still open

---

## 4. Scope of CLI Modifications

The CLI app must modify or generate the following classes of outputs:

## 5. New Audit Model

## 5.1 Replace status values

The CLI must replace coarse statuses like `found` with more precise states.

### Old

* found
* partial
* missing

### New

* `candidate_found`
* `implementation_verified`
* `dominant_but_contested`
* `closure_not_proven`
* `missing`

### Meaning

* `candidate_found`: file exists and plausibly matches responsibility
* `implementation_verified`: file exists, imports resolve, role fits, and execution path is validated
* `dominant_but_contested`: current runtime nucleus uses it, but competing or broken alternatives still exist
* `closure_not_proven`: strong file candidate exists but gap cannot be treated as closed
* `missing`: no adequate implementation found

This directly addresses the reviewâ€™s criticism that discovery and closure were conflated. 

---

## 5.2 Split confidence into two fields

The CLI must emit:

* `discovery_confidence`
* `canonicality_confidence`

### discovery_confidence

How confident the CLI is that the file exists and matches the described role

### canonicality_confidence

How confident the CLI is that this file is the correct authoritative implementation for the role

A file can have high discovery confidence and low canonicality confidence. This distinction is required. 

---

## 5.3 Add execution-verification fields

For every recommended authority, add:

```json
"execution_verification": {
  "imports_resolve": "yes | no | unknown",
  "runtime_exec_verified": "yes | no | unknown",
  "used_by_dominant_orchestrator": "yes | no | unknown",
  "test_reference_found": "yes | no | unknown",
  "contract_aligned": "yes | no | unknown",
  "duplicate_shadow_absent": "yes | no | unknown"
}
```

These fields were identified as the missing bridge between discovery and proof. 

---

## 5.4 Fix final assessment model

Replace final booleans with this structure:

```json
"final_assessment": {
  "has_dominant_runtime_nucleus": true,
  "has_uncontested_canonical_runtime": false,
  "has_multiple_competing_orchestrators": true,

  "has_single_proven_registry_mutation_authority": false,
  "has_single_proven_patch_pipeline": false

```

This removes the contradiction noted in the review.

## 7. Blocker Remediation Plan

The CLI must explicitly inspect and classify blocker defects.

## 7.1 Missing `ID_SCRIPT_INVENTORY.jsonl`

The audit states the pre-commit canonicality gate depends on this file and will fail without it. 

### CLI action

* search repo root for current inventory file
* search backups for the referenced file
* verify whether an updated replacement exists
* determine whether the gateâ€™s expected filename/path is stale
* classify one of:

  * `restore_needed`
  * `path_update_needed`
  * `gate_disable_needed`
  * `replacement_inventory_found`

### Output

A remediation recommendation such as:

* restore file from backup to canonical path
* patch gate to use replacement inventory
* disable gate until restored
* mark gate as broken and non-authoritative

---

## 7.2 `WRITE_POLICY_IDPKG.yaml` mismatch

The audit says `idpkg_runtime.py` expects this name, but only `UNIFIED_SSOT_REGISTRY_WRITE_POLICY.yaml` was found. 

### CLI action

* inspect loader code
* verify exact expected path/name
* verify whether the existing write policy has equivalent structure
* determine whether rename, alias, or code patch is safest

### Remediation states

* `rename_file_to_expected`
* `patch_loader_to_actual_filename`
* `create_alias_wrapper`
* `contract_source_missing`

---

## 7.3 Broken import mismatch

The audit found import mismatches such as reconciler ID mismatches and missing handler versions. 

### CLI action

For each broken import:

* locate exact imported symbol
* search for real file providing equivalent implementation
* compare API shape
* determine whether caller patch is valid
* determine whether imported file is legacy, moved, renamed, or actually missing

### Phase 9 â€” Final closure report

## 17. Non-Negotiable Rules

The CLI must obey these rules:

* do not treat plausible discovery as closure
* do not call a file canonical unless execution and authority checks support it
* do not merge competing architectures into one fake solution
* do not suppress broken dependencies
* do not infer that missing support files are harmless
* do not mark single-writer semantics proven unless they are actually proven
* do not call the runtime canonical if orchestrator competition remains unresolved
