  REGISTRY_file.json (Track A phases 1-6, Track B's registry shape
  reconciliation) need the registry file in the worktree's allowed_paths
   with an explicit merge strategy — the template's default squash-merge
   must be verified safe for registry JSON.

  Net Assessment

  The adaptation is feasible and the template adds real value here:

  - GATE-015 (Rollback Completeness) enforces the rollback_manifest
  requirement already planned
  - GATE-010 (Wiring Proof) forces the 9 solution-doc output files to
  have declared producers and consumers — this closes the "outputs exist
   as aspirations but not as artifacts" gap identified earlier
  - The phase_dag enforces the dependency: Track B's DQ answers must be
  recorded before Track A Phase 1 can execute
  - Step-level contracts force vague remediation steps ("verify imports
  resolve") into actual executable postconditions

  The main effort is the custom failure types and translating the Track
  B verification steps (which are currently described in prose in the
  solution doc) into executable shell commands for step
  preconditions/postconditions. That translation is the hardest part —
  it requires knowing what scripts actually exist to run those checks.
