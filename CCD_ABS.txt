  The system establishes a **Path Abstraction & Indirection Layer**
  designed to decouple your code from the physical file system. By
  using this system, you stop referencing files by their literal paths
  and start referencing them by **stable, semantic keys**.

 

  ## How it prevents import/path breaks

  The core of the strategy is moving the "knowledge" of where a file
  lives out of your scripts and into a central **Path Registry**.

  ### 1. The "Single Source of Truth" (SSOT)
  Instead of updating 50 scripts when a file moves, you update one
  entry in `config/path_index.yaml`.
  * **The Key**: A unique ID like `phase_docs.ph02_state_layer_spec`.
  * **The Value**: The actual path, e.g., `"spec/phase_docs/PH-
  02_State_Layer.md"`.

  ### 2. The Resolver Mechanism
  Your scripts use a **Path Resolver** (Python library or CLI) to find
  the file at runtime:

  ```python
  # Instead of: open("old/path/file.md")
  # Use this:
  from path_registry import resolve_path
  path = resolve_path("phase_docs.ph02_state_layer_spec")
  ```

  If the file moves, the **Key** remains exactly the same. The code
  doesn't break because the `resolve_path` function fetches the updated
  location from the registry.

  ---

  ## When to use this to fix "Breaks"

  According to the **UET & Path Abstraction SSOT**, you should
  prioritize migrating to this system in these scenarios:

  | Scenario | Use Path Abstraction? | Why? |
  | :--- | :--- | :--- |
  | **Cross-module boundaries** | **Mandatory** | Prevents one module's
  folder change from breaking another module. |
  | **High-churn files** | **Recommended** | If docs or specs move
  often, the registry absorbs the change. |
  | **Shared automation** | **Mandatory** | Ensures CLI tools and
  agents (like Aider or Codex) find files reliably. |
  | **Single local module** | **Optional** | If the impact is only
  local and stable, direct paths are okay. |

  Use It For
  Use path abstraction for these:

  - docs
  - schemas
  - configs
  - registry files
  - evidence paths
  - generated artifacts
  - CLI script entrypoints
  - executor/plugin locations
  - high-churn files and folders

  Example:

  schema_path = resolve_path("registry.schema.v3")
  policy_path = resolve_path("id.prefix_policy")
  gate_spec = resolve_path("guardrails.gate_catalog")

  Then moving the file means updating one path-index entry.

  Do Not Use It As The Primary Fix For
  Do not make regular Python imports depend on path registry unless the
  file is truly a plugin/executor loaded dynamically.

  Avoid this as the normal pattern:

  mod = import_module("ID_ALLOCATOR")
  allocate = mod.allocate_single_id

  That is acceptable for registered executors, but it is weaker for
  normal code because static analysis, refactors, type checking, test
  discovery, and import errors become harder to reason about.

  For normal Python modules, prefer:

  from govreg.id.allocation import allocate_single_id

  Then keep govreg.id.allocation stable, even if the implementation file
  behind it changes.

  Recommended Policy
  For existing files, stop making filename ID changes mandatory.

  Use this split instead:

  Existing imported source files:
    Keep readable, import-stable module names.
    Store stable_id in registry/manifest or a top-of-file metadata
  block.

  Executor scripts:
    May be referenced by executor_id/path_key.
    Can be dynamically resolved.

  Docs/config/schemas:
    Use stable semantic path keys.

  Directories:
    Use .dir_id anchors, but avoid making directory names carry
  identity.

  That means a file can have a stable identity without its filename
  becoming the identity.

  Example registry entry:

  paths:
    id.allocator:
      stable_id: "01999000042260124031"
      kind: "python_module"
      path: "01260207201000001250_REGISTRY/ID/1_runtime/allocators/
  unified_id_allocator.py"
      public_import: "govreg.id.allocation"
      aliases:
        - "ID_ALLOCATOR"

  Then code imports the public API:

  from govreg.id.allocation import allocate_single_id

  And automation resolves the artifact:

  resolve_path("id.allocator")

  Best Migration Path

  1. Freeze existing import-bearing filenames. No more mass renames for
     stable IDs.
  2. Add a central path_index.yaml or consolidate the existing ones.
  3. Implement a small path_registry.py if the current one is missing.
  4. Add stable IDs as metadata in the path registry, not as mandatory
     filename prefixes.
  5. Create compatibility shims only where renames already happened.
  6. Use a hardcoded-path scan to replace only high-risk paths first.
  7. Add a gate: “new important resources must have a path key,” but
     don’t fail the whole legacy codebase immediately.

  Bottom Line
  Yes, this approach is exactly the right direction for reducing rework
  from moves and renames. But the strongest version is:

  path registry for locating artifacts
  executor registry for dynamic tools
  stable package API for Python imports
  stable ID registry for identity

  Using stable IDs directly in existing filenames is the part causing
  churn. Treat IDs as metadata/registry identity, not as the physical
  import path.
